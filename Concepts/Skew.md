# Skewed Data — Root Cause Analysis

## 1. Intuition First

Most candidates jump straight to "log transform it." That's a **fix**, not a **diagnosis**. The interview-grade answer starts with: *skew is a symptom — what's the underlying data-generating process causing it?* The right fix depends entirely on the root cause; the wrong fix (e.g., log-transforming a bimodal mixture) can actively hurt the model.



---

## 2. The Root Cause Taxonomy

### A. Natural multiplicative / generative process (genuine skew — don't "fix," transform intentionally)
Many real-world quantities are generated by **multiplicative**, not additive, processes — income, city population, word frequency, file sizes, viral content views. Multiplicative processes produce **log-normal** distributions by the Central Limit Theorem applied in log-space.

**Diagnostic signature:** plot `log(x)` — if it looks roughly normal/symmetric, the skew is log-normal and structural, not an error.

### B. Bounded/floor-effect variables (structural, not an error)
Variables bounded at zero with no natural upper bound — counts, wait times, durations, sales amounts — mechanically produce right skew because they can't go negative but can grow arbitrarily large.

**Diagnostic signature:** check domain — is zero a hard floor? Does the variable represent a count, duration, or amount?

### C. Mixture of subpopulations (the most commonly missed root cause)
Data actually comes from **two or more distinct groups** with different centers, and pooling them creates apparent skew or bimodality that *looks* like a long tail but is structurally different.

**Example:** "session duration" pooling casual browsers (short sessions) and power users (long sessions) — the pooled histogram looks right-skewed, but it's really two overlapping normal-ish distributions, not one skewed one.

**Diagnostic signature:** segment by a plausible categorical variable (user tier, device, region, cohort) and replot — if the skew disappears or splits into two clean modes per segment, this is the root cause. **A log transform does NOT fix this** — it just relabels the same two-population problem in log-space.

### D. Outliers from genuine rare extreme events (not errors — real tail risk)
Some skew comes from real rare large-magnitude events: a whale customer, a viral post, a genuine fraud ring. These are legitimate data points, not noise.

**Diagnostic signature:** are the extreme values *plausible* given domain knowledge and business context? Cross-check against known caps (e.g., is a $50M single transaction physically possible for this business, or impossible)?

### E. Sampling / selection bias (censoring, truncation, survivorship)
Skew can be an artifact of **how the sample was collected**, not the true underlying population.

**Examples:**
- Survivorship bias: only observing outcomes of things that "survived" (e.g., only completed transactions, only users who didn't churn before day 30)
- Truncation: values below/above a threshold were never recorded (e.g., loan applications below a minimum amount aren't logged)
- Right-censoring: event hasn't happened yet at data-collection time (common in survival/duration data — "time to churn" for still-active users)

**Diagnostic signature:** check the data collection pipeline — is there an implicit filter (only successful records, only records above/below some system threshold)? This root cause needs **survival analysis / inverse-probability weighting**, not a transform.

### F. Measurement / instrument artifacts
Rounding, sensor saturation/clipping, minimum billing units, or floor/ceiling effects in how a value was recorded (e.g., "99+" bucket in a survey, sensor maxing out at a hard limit).

**Diagnostic signature:** look for suspicious **spikes at round numbers or at the exact min/max** of the observed range — a real distribution rarely stacks values exactly at a boundary.

### G. Data entry / pipeline errors
Duplicate zeros, unit mismatches (some rows in cents, some in dollars), copy-paste errors, string-to-number parsing bugs producing extreme values.

**Diagnostic signature:** the extreme values are **implausible** given domain knowledge (negative age, a "duration" of 10 million seconds for a web session) — this is the one case where you should **clean/remove**, not transform.

## 3. Diagnostic Workflow (how to actually run this in practice)

1. **Plot the raw histogram + boxplot** — get the shape and see obvious outliers
2. **Plot `log(x)` histogram** — does it become symmetric? → points to (A) multiplicative process
3. **Check the variable's domain and floor** — is zero a hard bound? → points to (B)
4. **Segment by every plausible categorical variable and replot** — does skew resolve into clean modes? → points to (C), the most-missed cause
5. **Check skewness/kurtosis stats** alongside the plots — skewness alone doesn't distinguish (A) from (C), you need the segmented view
6. **Sanity-check extreme values against domain limits** — plausible-but-rare (D) vs impossible (G)
7. **Trace the data pipeline** — any filters, thresholds, or "not yet observed" cutoffs upstream? → points to (E)
8. **Look for suspicious spikes at round numbers or exact min/max** → points to (F)

---

## 4. Root Cause → Correct Action (mapping table)

| Root Cause | Correct Action | Wrong Action |
|---|---|---|
| (A) Multiplicative/log-normal | Log transform (this is the case it's actually meant for) | Leaving as-is for a linear model |
| (B) Bounded/count/duration | Log or sqrt transform, or a count-appropriate model (Poisson/Negative Binomial) | Assuming normality, using vanilla linear regression untransformed |
| (C) Mixture of subpopulations | **Segment the model** (separate models per group, or add group as a feature/interaction) | Log-transforming — masks the real structure |
| (D) Genuine rare extreme events | Keep the data; consider robust loss (Huber) or a model that handles tails (tree-based, quantile regression) | Deleting real outliers — throws away real signal |
| (E) Sampling/selection bias | Survival analysis, inverse-probability weighting, or redesign collection | Any transform — doesn't address biased sampling at all |
| (F) Measurement artifact | Correct/re-bucket at the source if possible, or model the censored boundary explicitly (Tobit-style) | Transform without acknowledging the artificial boundary |
| (G) Data entry error | Clean/fix/remove at the source | Log-transform to "smooth over" an actual bug |

## 5. Worked Example

**Scenario:** "Order value" is heavily right-skewed in an e-commerce dataset.

- Step 1: log(order value) histogram is *almost* symmetric but has a **second small bump** at very high values.
- Step 2: Segment by `customer_type` (retail vs wholesale). Retail is cleanly log-normal. Wholesale accounts for the second bump and has an entirely different scale.
- **Root cause identified: (C) mixture of subpopulations**, not pure (A).
- **Correct action:** model retail and wholesale separately (or include `customer_type` as a strong interacting feature), rather than a single log-transform applied to the pooled data, which would blur the two populations together and bias predictions for both.

---

## 6. Interview Trap (the one to say out loud unprompted)

**"Skewness itself doesn't tell you the root cause — the same skewness statistic can come from a genuine log-normal process (A) or from a mixture of two populations (C), and they require opposite fixes."** Log-transforming a mixture problem doesn't fix it — it just makes the mixture harder to see. This is the single most L5-differentiating point in this entire topic: **most people go straight from "skewed" to "log transform," skipping the diagnosis that determines whether that's even the right move.**

## Comprehension Check

**Q1.** Two datasets have identical skewness values. One is genuinely log-normal, one is a mixture of two subpopulations. What single diagnostic step distinguishes them?
**Q2.** Why doesn't a log transform fix skew caused by survivorship bias?
**Q3.** You see a spike of values exactly at 999999 in a "session duration in seconds" column. Which root cause is this, and what should you do?
**Q4.** A skewed "time to churn" variable includes many still-active users. What's the root cause category, and what technique addresses it (not a transform)?
Q1. Plot the conditional distributions after segmenting by a plausible categorical variable (tier, cohort, region). A genuine log-normal stays skewed within every segment. A mixture splits into two symmetric-ish humps — one per subpopulation — and the pooled skew largely disappears. Skewness is a single-number summary that's blind to this structure; the segmented histogram exposes it directly.
Q2. Survivorship bias removes the left tail entirely — the low values (failed companies, dropped users, short sessions) simply don't exist in the dataset. A log transform compresses the right tail but does nothing to restore the missing left tail. The distribution is truncated at the bottom, not stretched at the top. The fix is upstream: reconstruct or acknowledge the missing population, not rescale the one you have.
Q3. That's a sentinel value / data quality issue — 999999 is a system placeholder (cap, timeout, or "still active" flag encoded as a number) masquerading as a real measurement. What to do: treat it as missing, not as data. Null it out, then decide separately whether those rows represent censored observations (still-active sessions → survival analysis) or errors (timeout artifacts → drop). Never include it in a distribution or summary stat as-is.
Q4. The root cause is censoring — right-censored observations where the event (churn) hasn't occurred yet, so the true duration is unknown and bounded only from below. The technique is survival analysis (Kaplan-Meier for description, Cox proportional hazards or parametric survival models for inference). These methods explicitly model the censoring mechanism rather than ignoring it or imputing it. Transforming the time variable and running a standard regression is wrong here because it treats censored values as if they were actual churn times, which biases every estimate.

## Multiplicative Processes and Log-Normal Distributions

The core idea is a direct parallel to why *additive* processes produce normal distributions — just one level of abstraction up.

---

### Start with the familiar: Additive CLT

The Central Limit Theorem says: if you **add** many independent random variables, the result converges to a **normal distribution**, regardless of the individual distributions.

$$X = x_1 + x_2 + x_3 + \cdots + x_n \xrightarrow{n \to \infty} \mathcal{N}(\mu, \sigma^2)$$

This is why height, measurement error, and test scores are bell-shaped — they're sums of many small independent effects.

---

### Now: Multiplicative processes

Many real quantities grow by **proportional factors** at each step. For example, a company's revenue this year is last year's revenue times some growth factor:

$$X = x_1 \cdot x_2 \cdot x_3 \cdots x_n$$

Each $x_i$ is a random multiplier (say, growth rate, share factor, probability of going viral at each step).

---

### The key move: take the log

$$\ln(X) = \ln(x_1) + \ln(x_2) + \ln(x_3) + \cdots + \ln(x_n)$$

A **product becomes a sum** under the logarithm. Now the CLT kicks in on $\ln(X)$: the sum of many independent log-terms converges to a normal distribution.

So:

$$\ln(X) \sim \mathcal{N}(\mu, \sigma^2)$$

which means $X$ itself follows a **log-normal distribution**.

---

### Why this matches the real world

| Quantity | Multiplicative mechanism |
|---|---|
| Income | Each raise/promotion is a % increase on current salary |
| City population | Each year's growth is a fraction of current population |
| Word frequency | Each word's usage spreads proportionally to existing usage |
| File sizes | Files are built by compounding layers (headers, compression, encoding) |
| Viral views | Each share multiplies from current reach |

In every case, the "shock" at each stage is **proportional to the current value**, not a fixed additive amount.

---

### What log-normal distributions look like

This produces a characteristic shape:

- **Right-skewed** with a long tail (a few cities are enormous; most are tiny)
- **Symmetric on a log scale** — if you plot $\ln(X)$, you get a bell curve
- **Multiplicative mean** — the natural average is the geometric mean, not the arithmetic mean

This is also why using the *arithmetic* mean for income is misleading — a billionaire can pull it far from the typical person's experience, but the *median* (which corresponds to the geometric mean on a log-normal) is far more representative.

---

### The one-line summary

> The CLT says sums of random variables go normal. Multiplicative processes are sums *in log-space*. So their logarithms go normal — making the original quantities log-normal.

It's the same theorem, applied after a change of variables.
