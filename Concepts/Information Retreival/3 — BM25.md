# Chapter 3 — BM25

## What is it?

BM25 (Best Match 25) is a ranking function that improves directly on TF-IDF. It was developed in the 1990s as part of the Okapi research project at City University London, and it remains the **default ranking algorithm in Elasticsearch, Lucene, and Solr today** — thirty years later. If a company says "we use keyword search," they almost certainly mean BM25 under the hood.

The "25" in BM25 is just a version number — it was the 25th iteration of the Best Match experiments. BM1 through BM24 didn't make the cut.

---

## The intuition

TF-IDF has two problems BM25 fixes directly:

**Problem 1 — TF grows without bound.** In TF-IDF, if term "neural" appears 100 times in a doc it scores 100× higher than a doc where it appears once. But does mentioning something 100 times make a document 100× more relevant? No. After a handful of occurrences you've established the topic — more repetitions add diminishing signal. BM25 **saturates TF** — the score keeps growing but slower and slower, approaching a ceiling.

**Problem 2 — Length normalization is crude in TF-IDF.** TF-IDF divides by document length, which fully normalizes away length. But a long document covering a topic extensively is genuinely more relevant than a short one that mentions it once. You want partial length normalization — penalize unnecessarily long documents, but don't fully erase the advantage of comprehensive coverage. BM25 gives you a **tunable knob** for how much length matters.

---

## The equation

```
score(q, d) = Σ IDF(t) × [ TF(t,d) × (k₁ + 1) ] / [ TF(t,d) + k₁ × (1 - b + b × |d|/avgdl) ]
```

Summed over each term `t` in query `q`.

Breaking down every symbol:

| Symbol | Meaning | Typical value |
|--------|---------|---------------|
| `TF(t,d)` | Raw count of term t in document d | any integer ≥ 0 |
| `IDF(t)` | Same as TF-IDF: `log(N / df(t))` | depends on corpus |
| `k₁` | TF saturation parameter | 1.2 to 2.0 |
| `b` | Length normalization strength | 0.75 |
| `\|d\|` | Length of document d (word count) | varies |
| `avgdl` | Average document length across corpus | varies |

**What k₁ controls:**
```
k₁ = 0   → TF completely ignored, score = IDF only
k₁ = 1.2 → fast saturation (default for short docs)
k₁ = 2.0 → slower saturation (default for longer docs)
k₁ → ∞   → back to linear TF, no saturation (= TF-IDF behaviour)
```

**What b controls:**
```
b = 0    → document length ignored entirely
b = 0.75 → partial normalization (standard default)
b = 1.0  → full length normalization (= TF-IDF behaviour)
```

**The saturation mechanism — why the fraction works:**

Look at the numerator and denominator separately:

```
numerator:   TF × (k₁ + 1)      grows linearly with TF
denominator: TF + k₁ × (...)    also grows linearly with TF
```

As TF → ∞, the fraction → (k₁ + 1) / 1 = k₁ + 1. The score hits a ceiling. No matter how many times a term appears, it can never exceed this cap. That ceiling is what "saturation" means.

## Worked numeric example

```
corpus:
  D1 = "the cat sat on the mat"         → |D1| = 6
  D2 = "cat cat cat cat cat on the mat" → |D2| = 8
  D3 = "the mat and the hat"            → |D3| = 5

N = 3
avgdl = (6 + 8 + 5) / 3 = 6.33

query: "cat"
k₁ = 1.5, b = 0.75
```

### Step 1 — compute IDF("cat")

```
df("cat") = 2   (D1 and D2 contain "cat")
IDF("cat") = log(3 / 2) = log(1.5) = 0.405
```

### Step 2 — compute the length normalization factor for each doc

```
norm(D1) = 1 - b + b × |D1|/avgdl = 1 - 0.75 + 0.75 × (6/6.33)
         = 0.25 + 0.75 × 0.948
         = 0.25 + 0.711 = 0.961

norm(D2) = 1 - 0.75 + 0.75 × (8/6.33)
         = 0.25 + 0.75 × 1.264
         = 0.25 + 0.948 = 1.198   ← penalized for being longer

norm(D3) = 1 - 0.75 + 0.75 × (5/6.33)
         = 0.25 + 0.75 × 0.790
         = 0.25 + 0.592 = 0.842   ← rewarded for being shorter
```

### Step 3 — compute TF("cat") for each doc

```
TF("cat", D1) = 1   (appears once)
TF("cat", D2) = 5   (appears five times)
TF("cat", D3) = 0   (doesn't appear)
```

### Step 4 — compute BM25 score for each doc

```
score(D1) = 0.405 × [1 × (1.5 + 1)] / [1 + 1.5 × 0.961]
          = 0.405 × 2.5 / [1 + 1.441]
          = 0.405 × 2.5 / 2.441
          = 0.405 × 1.024
          = 0.415

score(D2) = 0.405 × [5 × 2.5] / [5 + 1.5 × 1.198]
          = 0.405 × 12.5 / [5 + 1.797]
          = 0.405 × 12.5 / 6.797
          = 0.405 × 1.839
          = 0.745

score(D3) = 0.000   (TF = 0, so whole expression = 0)
```

### Step 5 — compare with what TF-IDF would give

```
TFIDF(D1) = (1/6)  × 0.405 = 0.068
TFIDF(D2) = (5/8)  × 0.405 = 0.253
TFIDF(D3) = 0

BM25 ratio D2/D1:   0.745 / 0.415 = 1.79×
TFIDF ratio D2/D1:  0.253 / 0.068 = 3.72×
```

D2 has 5× the term frequency of D1. TF-IDF gives it 3.72× the score — nearly proportional to the raw count. BM25 gives it only 1.79× — it acknowledges D2 is more relevant, but not nearly as dramatically. That's saturation working correctly.

## Visualizing TF saturation

As raw TF count increases from 1 to 30, here is how BM25 score grows vs TF-IDF (holding IDF and length constant, k₁=1.5):

```
TF count  |  BM25 score  |  TF-IDF score
----------|--------------|---------------
1         |  0.415       |  0.068
2         |  0.558       |  0.135
5         |  0.745       |  0.338
10        |  0.849       |  0.675
20        |  0.920       |  1.350
30        |  0.946       |  2.025   ← TF-IDF keeps growing, BM25 flattens
∞         |  1.012       |  ∞
```

BM25 approaches a ceiling (~1.01 here). TF-IDF has no ceiling.

## Why it works / why it fails

**Why it works:**
- TF saturation is empirically correct — relevance does plateau with more occurrences
- Tunable parameters mean you can adapt to your corpus (short tweets vs long articles need different k₁)
- Length normalization is partial, not total — comprehensive documents aren't unfairly penalized
- Still fast — same inverted index infrastructure as TF-IDF, just a different scoring formula
- Battle-tested for 30 years across every type of text corpus imaginable

**Why it fails:**
- Still bag-of-words — word order, syntax, and semantics are completely ignored
- Still vocabulary-dependent — "car" and "automobile" are different terms
- Parameters k₁ and b need tuning per corpus — wrong values hurt performance
- Rare terms in very small corpora get artificially inflated IDF scores
- No way to handle multi-word concepts — "New York" is treated as two separate terms "new" and "york"

---

## The one thing to remember

BM25 is TF-IDF with two fixes: **TF saturates** (diminishing returns on repeated terms) and **length normalization is tunable** (b controls how much document length matters). These two changes are why it's still the default keyword search algorithm 30 years later.

## Formulas used in this chapter

| Formula | Meaning |
|---------|---------|
| `score(q,d) = Σ IDF(t) × [TF(t,d)×(k₁+1)] / [TF(t,d) + k₁×(1-b+b×\|d\|/avgdl)]` | Full BM25 scoring formula |
| `IDF(t) = log(N / df(t))` | Same IDF as TF-IDF |
| `norm(d) = 1 - b + b × \|d\| / avgdl` | Length normalization factor — approaches 1 when doc length = average |
| `TF ceiling = k₁ + 1` | Maximum contribution of TF regardless of raw count — the saturation cap |
| `b = 0 → no length norm` | Disables length normalization entirely |
| `b = 1 → full length norm` | Same as TF-IDF normalization |

## Interview Q&A

**Q1. What are the two main improvements BM25 makes over TF-IDF?**

First, TF saturation via the k₁ parameter — the scoring formula asymptotically approaches a ceiling as TF grows, so a term appearing 100 times doesn't score 100× higher than one appearing once. Second, tunable length normalization via the b parameter — rather than fully dividing by document length (TF-IDF) or ignoring it entirely, b lets you choose how much length should penalize a document, with 0.75 as the empirically validated default.

**Q2. What happens to BM25 when k₁ = 0?**

The TF component disappears entirely. The numerator becomes `0 × (0+1) = 0`... wait, let's be precise: when k₁=0, the formula becomes `IDF × [TF × 1] / [TF + 0] = IDF × 1 = IDF`. Every document that contains the term gets exactly the IDF score regardless of how many times the term appears. It becomes a pure IDF ranker — presence matters, frequency doesn't. Useful when documents are very short (tweets, titles) where TF is not meaningful.

**Q3. Why is b = 0.75 the default and not 0 or 1?**

It's empirically validated across decades of TREC benchmarks (the academic IR competitions). b=0 ignores length entirely, which hurts because spammy short docs full of keywords score too high. b=1 fully normalizes by length, which hurts because it treats a 10-page comprehensive document the same as a 1-sentence document that mentions the term once. 0.75 sits in the sweet spot — it penalizes documents longer than average but doesn't fully erase the advantage of comprehensive coverage.

**Q4. BM25 is 30 years old. Why is it still the default in Elasticsearch?**

Three reasons. First, it's extremely fast — it uses the same inverted index infrastructure as any other keyword search, with scoring being just an arithmetic formula applied at lookup time. Second, it works well — decades of benchmarks show it matches or beats more complex methods on keyword-heavy queries with little tuning. Third, it's interpretable — every score can be traced back to specific term frequencies and IDF values, which matters for debugging and auditing. Neural methods beat it on semantic tasks but require GPU inference, embedding storage, and ANN infrastructure. For exact keyword matching, BM25 is hard to beat at its cost.

**Q5. How would you tune k₁ and b for a corpus of Twitter posts (avg length: 15 words)?**

For very short documents, TF is barely meaningful — a term appearing twice in a 15-word tweet vs once is not a strong signal. So lower k₁ (closer to 0, say 0.5–1.0) to reduce TF's influence. For length normalization, since all documents are similarly short, length variation is minimal and normalization matters less — lower b (closer to 0, say 0.3–0.5) so short docs aren't over-penalized for the small natural variation in tweet length. In practice you'd tune both on a held-out labeled dataset using NDCG@k as the metric (Chapter 5).
