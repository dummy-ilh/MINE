# Chapter 11 — Retrieval Internals: Query Expansion, Pseudo-Relevance Feedback, and Field Weighting

## What is it?

Everything up through Chapter 10 assumed the query itself was fixed and good, and the only job was to score `score(q, d)` well. Chapter 11 attacks the other half of the problem: **the query is often broken, and the document has structure the scorer is ignoring.** These three techniques fix that before or during scoring, without touching the core ranking function.

---

## 11.1 Query Expansion

### The problem

A user types "car repair." The best document on earth says "automobile maintenance" and never uses the word "car" or "repair." Under TF-IDF/BM25 exact-match scoring, that document gets a score of zero for this query. The vocabulary mismatch problem from Chapter 2 strikes again.

### The idea

Before scoring, rewrite the query to include related terms:

```
original query:   "car repair"
expanded query:   "car repair automobile maintenance mechanic"
```

Now the exact-match scorer has a chance to hit the automobile-maintenance document too.

### Where the expansion terms come from

| Source | How it works | Weakness |
|---|---|---|
| Thesaurus / WordNet | Static synonym lookup ("car" → "automobile", "vehicle") | Misses domain slang, context-blind |
| Word embeddings (word2vec, GloVe) | Nearest neighbors in vector space | Can pull in related-but-wrong terms ("car" → "carpool", "carpet" by accident) |
| Click logs | Terms that co-occur in queries that led to clicks on the same doc | Needs traffic; cold-start problem for new queries |
| Pseudo-relevance feedback | See 11.2 below | Assumes top-k results are already decent |

### Weighted expansion

Expansion terms are almost never added with equal weight to the original terms — that would let noise overwhelm signal. A common formulation:

```
q_expanded = q_original + α · q_expansion_terms
```

Where `α < 1` (commonly 0.3–0.5) down-weights the expansion terms relative to the original query. If you set `α = 1`, an irrelevant expansion term can outvote the term the user actually typed.

### Why it works / why it fails

**Works:** directly attacks vocabulary mismatch, the single biggest weakness of exact-match retrieval (Chapter 2, "why it fails").

**Fails:**
- **Query drift** — expand "jaguar" (the car) and a thesaurus/embedding might add "cat", "predator", "Amazon rainforest" if the model doesn't know which sense of "jaguar" was meant. This is the classic **polysemy trap**.
- Expansion is a pre-processing step bolted onto exact-match retrieval; it's a patch, not a fix. Dense retrieval (Chapter 4) solves the same problem structurally, since "car" and "automobile" already live near each other in embedding space with no expansion step needed.

---

## 11.2 Pseudo-Relevance Feedback (PRF)

### The idea

Query expansion needs expansion terms from *somewhere*. PRF's trick: **assume your first-pass retrieval was roughly right**, then use the top-k results themselves as free labeled data.

```
Step 1: run the original query q, get top-k documents (commonly k = 10)
Step 2: pull the highest-TF-IDF/BM25 terms out of those k documents
Step 3: add those terms to q, weighted down (α from above)
Step 4: re-run retrieval with q_expanded, return the FINAL ranked list to the user
```

This is a **two-pass retrieval**: one silent pass to gather expansion terms, one visible pass that the user actually sees.

### Worked example

```
query: "jaguar speed"

Pass 1 — retrieve top 3 docs with original query:
  D1: "Jaguar F-Type top speed 186 mph, 0-60 in 3.5 seconds"
  D2: "Jaguar big cat can reach speeds of 50 mph in short bursts"
  D3: "Jaguar E-Type classic sports car speed record"

Pull high-TF-IDF terms from these 3 docs (excluding query terms already present):
  "F-Type" appears in D1, D3 (car-related)
  "mph" appears in all 3 (ambiguous — appears for both senses, gets picked up regardless)
  "sports" appears in D1, D3
  "big cat" appears only in D2

  → because 2 of 3 top docs are about the car, the extracted terms skew car-ward:
     expansion terms = {"F-Type", "sports", "mph"}

Pass 2 — re-run with:
  q_expanded = "jaguar speed" + 0.4·{"F-Type", "sports", "mph"}

  → D1 and D3 (car docs) now score even higher, D2 (animal doc) gets a small
    unrelated boost from "mph" but not enough to out-rank D1/D3.
```

Notice what just happened: PRF **self-reinforced whichever interpretation was already winning** in pass 1. That's both its strength (no external resource needed — free, corpus-native, always up to date) and its core failure mode.

### Why it works / why it fails

**Works:**
- No thesaurus or embedding model needed — the corpus expands its own queries
- Adapts automatically to any domain (legal, medical, code) without retraining anything
- Consistently improves recall in classical IR benchmarks (TREC) by a measurable margin

**Fails:**
- **Query drift, amplified.** If pass 1's top-k is dominated by an *irrelevant* interpretation (e.g., a spam-heavy corpus, or an ambiguous query that happens to skew toward the wrong sense), PRF doubles down on the wrong answer instead of correcting it. There is no human-in-the-loop check between pass 1 and pass 2.
- **2x latency** — every PRF query is two full retrieval passes instead of one. This matters a lot at web scale (Chapter 13).
- Sensitive to `k`: too small and you don't get enough signal; too large and you pull in noise from documents that were only marginally relevant in the first place.

## 11.3 Field Weighting / Document Re-weighting

### The problem

So far, every document has been treated as one flat bag of words. Real documents aren't flat — they have a **title**, a **body**, **metadata** (author, tags, URL), sometimes **anchor text** (the link text other pages use to link to this page). A term appearing in the title is a much stronger relevance signal than the same term buried in paragraph 40 of the body.

### The idea

Score each field separately, then combine with weights:

```
score(q, d) = w_title · score(q, d.title)
            + w_body  · score(q, d.body)
            + w_meta  · score(q, d.metadata)
```

Where `w_title + w_body + w_meta` don't need to sum to 1 — these are tunable boost factors, commonly `w_title ≈ 3–5×` the weight of `w_body`.

### Worked example

```
query: "neural network"

Doc A:
  title = "Introduction to Neural Networks"      → title score = 0.80
  body  = generic ML text, mentions term 3 times → body score  = 0.10

Doc B:
  title = "Cooking Recipes"                        → title score = 0.00
  body  = deep technical essay on neural networks,
          mentions term 40 times                    → body score  = 0.45

with weights w_title = 4, w_body = 1:

score(Doc A) = 4×0.80 + 1×0.10 = 3.30
score(Doc B) = 4×0.00 + 1×0.45 = 0.45

→ Doc A wins decisively, even though Doc B has far more raw term occurrences.
```

This matches intuition: a document *about* neural networks (says so right in the title) should outrank a document that merely *mentions* them 40 times in passing.

### Anchor text — a special case

Anchor text is the clickable text other pages use when linking to a page, e.g. `<a href="ibm.com">Big Blue</a>`. It's a field like any other, but it's written by *other authors*, not the page owner — which makes it resistant to keyword stuffing (you can't easily manipulate what other sites call you) and is one of the reasons early Google (PageRank + anchor text weighting) beat pure on-page TF-IDF engines.

### Why it works / why it fails

**Works:**
- Encodes cheap, high-precision structural priors ("titles matter more") without any ML
- Anchor text weighting specifically resists spam/keyword-stuffing better than body-only scoring
- Composable with everything else in this book — it's a wrapper around whatever `score()` you already have

**Fails:**
- Weights (`w_title`, `w_body`, ...) are usually hand-tuned or grid-searched, not learned — this is exactly the kind of decision Learning to Rank (Chapter 12) automates and does better by treating "which field matched" as a *feature* instead of a hard-coded constant.
- Doesn't help at all with vocabulary mismatch or word order — it's purely about *where* a match occurred, not *whether a better match exists elsewhere*.

| Formula | Meaning |
|---|---|
| `q_expanded = q + α·q_expansion` | Blend original query with expansion terms, down-weighted by α |
| PRF loop | retrieve top-k → extract high-weight terms → re-run query with expansion |
| `score(q,d) = Σ_f w_f · score(q, d.f)` | Weighted sum of per-field scores across fields f (title, body, metadata...) |

## Interview Q&A

**Q1. Why down-weight expansion terms instead of treating them equally with the original query?**

The original query terms are directly from the user and carry the highest confidence signal about intent. Expansion terms are inferred — by a thesaurus, embedding neighbor search, or PRF — and carry real risk of being wrong (wrong word sense, unrelated neighbor, drift toward the wrong topic). Weighting them at `α < 1` lets them add recall without letting a single bad expansion term outvote what the user actually typed.

**Q2. Walk through why PRF can make results worse, not better.**

PRF assumes the top-k documents from pass 1 are relevant, then extracts terms from them and re-queries. If the query is ambiguous (e.g. "jaguar") and pass 1's top-k happens to skew toward the wrong sense — say, spam or an unlucky ranking accident — PRF extracts terms *from the wrong interpretation* and the second pass amplifies that mistake instead of correcting it. There's no external check; it's a closed loop that trusts its own first guess.

**Q3. Why does anchor text resist manipulation better than body text?**

Anchor text is written by other site owners linking *to* a page, not by the page's own author. A spammer can stuff their own body text with any keyword they want, but they can't force other independent sites to describe them with the same spammy keywords. This makes anchor text a harder-to-game relevance signal, which is part of why early web search engines weighted it heavily.

**Q4. Give an example where field weighting alone is not enough, and what technique from a later chapter would help.**

Field weighting only changes *where* a match counts more, not *whether the best possible document was even retrieved*. If the best document uses "automobile" throughout (title and body) but the query says "car," field weighting doesn't help at all — the term literally never matches in any field. That's a vocabulary mismatch problem, and it needs query expansion (11.1) or dense retrieval (Chapter 4), not field re-weighting.

**Q5. Why is PRF roughly 2x latency compared to a normal query?**

PRF requires a full first-pass retrieval to obtain the top-k documents, then a second full retrieval with the expanded query to produce the final result the user sees. Both passes hit the index and run the full scoring pipeline, so end-to-end latency is approximately double a single-pass query — a real cost at web scale where p99 latency budgets are often single-digit milliseconds to a few hundred milliseconds.

# Chapter 12 — Learning to Rank (LTR)

## What is it?

Every scoring function so far — TF-IDF, BM25, field-weighted combinations — has been **hand-designed**. A human decided the formula, decided `w_title = 4`, decided to take a log here and a ratio there. Learning to Rank flips this: **instead of hand-designing `score(q,d)`, train a model to learn it from data**, where the data is real user behavior (clicks, purchases, human relevance judgments).

This is the single most consequential idea in modern search ranking. Every major search/recommendation system at scale (Google, Bing, Amazon, YouTube, LinkedIn feed) has an LTR model — usually a gradient-boosted tree ensemble or neural ranker — sitting downstream of a first-pass retriever like BM25 or dense retrieval.

---

## 12.1 Why not just use TF-IDF/BM25 scores directly?

Because relevance depends on **dozens of signals**, not one formula:
- BM25 text-match score
- PageRank / authority score
- Click-through rate (CTR) history for this query-document pair
- Freshness (how recently was this published)
- Whether the query term matched in the title vs body
- User's personal click history
- Document length, load time, mobile-friendliness...

No human can hand-tune the right combination of 50+ signals with correct interaction effects. LTR treats this as what it actually is: **a supervised machine learning problem.**

---

## 12.2 The three paradigms

The core design decision in LTR is: **what does the model's loss function compare?**

### Pointwise

Treat ranking as a **regression or classification problem, one document at a time.** The model predicts a relevance score `ŷ = f(q, d)` for a single (query, document) pair, and is trained against a ground-truth label (e.g. 0=irrelevant, 1=relevant, 2=highly relevant) using ordinary regression loss (MSE) or classification loss (cross-entropy).

```
loss = Σ over all (q,d) pairs:  (ŷ(q,d) - y(q,d))²
```

**Weakness:** the model never sees that documents are being *compared to each other* for the same query. It might get every individual score roughly right in absolute terms, yet still produce the wrong *relative order* — and relative order is the only thing that actually matters for ranking.

### Pairwise

Treat ranking as **"which of these two documents should rank higher"** — a classification problem over *pairs* of documents for the same query. If document A is more relevant than document B for query q, the loss penalizes the model whenever it scores B ≥ A.

```
loss = Σ over all pairs (d_i, d_j) where d_i should rank above d_j:
       penalty if  f(q, d_i) ≤ f(q, d_j)
```

This is what **RankNet** does (12.3). It's a big improvement over pointwise because it directly optimizes for relative order, which is what ranking actually needs.

**Weakness:** all pairwise mistakes are penalized equally, regardless of *where in the ranked list* they occur. Swapping documents ranked #1 and #2 is a catastrophe for user experience (users barely scroll past the top 3 results); swapping documents ranked #47 and #48 is essentially invisible to any real user. Pairwise loss treats both mistakes as the same size of error.

### Listwise

Treat ranking as **optimizing the entire ordered list at once**, directly against a ranking metric like NDCG (Normalized Discounted Cumulative Gain) that already knows position matters — an error at position 1 costs far more than an error at position 40.

```
loss ≈ − NDCG(predicted_list, ideal_list)     (directly, or via a differentiable proxy)
```

**LambdaMART** (12.3) is the standard listwise-flavored approach used in production.

### Summary table

| Paradigm | Unit of comparison | Loss cares about position? | Example model |
|---|---|---|---|
| Pointwise | 1 document | No | Simple regression/classification |
| Pairwise | 2 documents | No | RankNet |
| Listwise | entire ranked list | **Yes** | LambdaRank, LambdaMART |

---

## 12.3 RankNet, LambdaRank, LambdaMART

These three form an evolutionary chain — each fixes the previous one's biggest weakness.

### RankNet (pairwise)

For a pair of documents `(i, j)` where `i` should rank above `j`, RankNet computes the probability that the model believes `i` should outrank `j`:

```
P(i > j) = σ(f(q,i) − f(q,j))          [σ = sigmoid]
```

and trains with cross-entropy loss against the true label (i should outrank j). The gradient of this loss with respect to the model's scores is what actually gets backpropagated — call this gradient `λ_ij`.

**Problem:** RankNet's gradient `λ_ij` has the same magnitude whether the pair `(i,j)` is at positions (1,2) or (49,50) in the ranked list. It doesn't know position exists.

### LambdaRank (pairwise, position-aware)

**The one-line fix that made this whole family famous:** multiply RankNet's gradient by `|ΔNDCG|` — how much the ranking metric (NDCG) *would* change if you swapped `i` and `j`.

```
λ_ij(LambdaRank) = λ_ij(RankNet) × |ΔNDCG_ij|
```

Swapping positions #1 and #2 produces a large `ΔNDCG` (because NDCG heavily discounts by position — see the discount term `1/log₂(rank+1)`), so that mistake gets a big gradient and gets corrected hard during training. Swapping #47 and #48 produces a tiny `ΔNDCG`, so the model barely bothers fixing it. **This is exactly the fix pairwise RankNet was missing** — it makes the pairwise loss position-aware without needing NDCG to be directly differentiable (NDCG itself, because of sorting, isn't a differentiable function — you can't backprop through it directly. LambdaRank sidesteps this by defining the *gradient* directly rather than deriving it from a loss function.).

### LambdaMART (LambdaRank + gradient-boosted trees)

LambdaMART = LambdaRank's `λ` gradients, used to train **gradient-boosted regression trees (MART = Multiple Additive Regression Trees)** instead of a neural net.

Why trees, specifically, for ranking? Because ranking features are famously heterogeneous and non-smooth: BM25 score, PageRank, CTR, freshness-in-days, is-mobile (boolean)... Tree ensembles handle this mix of continuous/discrete/skewed features, missing values, and nonlinear interactions (e.g. "freshness only matters if the query is about news") far more gracefully than a linear model or a naively-designed neural net, and they're the reason LambdaMART became — and largely remains — the industry-standard production LTR model (it won the Yahoo! Learning to Rank Challenge and is what most production search-ranking stacks are built on or benchmarked against).

### The chain, visually

```
RankNet:      pairwise, position-blind
                    │  (multiply gradient by |ΔNDCG|)
                    ▼
LambdaRank:   pairwise, position-AWARE
                    │  (swap neural net for gradient-boosted trees)
                    ▼
LambdaMART:   position-aware + tree-based → production standard
```

---

## 12.4 Feature engineering for ranking

LTR models are only as good as their input features. A typical production feature vector for one `(query, document)` pair might include:

| Feature category | Examples |
|---|---|
| Text match | BM25 score, TF-IDF score, exact title match (boolean), query-term coverage % |
| Authority | PageRank, domain authority, number of inbound links |
| Behavioral | Historical CTR for this (query, doc) pair, dwell time, bounce rate |
| Freshness | Days since published, days since last updated |
| Query-independent | Document length, spam score, mobile-friendliness |
| Personalization | User's click history on similar docs, user's location match |

Notice: **BM25 score itself becomes just one input feature among dozens**, not the final answer. This is the key mental shift of this chapter — everything in Chapters 2–3 didn't get thrown away, it got *demoted from "the score" to "one feature the real model uses."*

### Worked mini-example (pointwise, for intuition)

```
Query: "gradient boosting"

Doc A features: [BM25=8.2, PageRank=0.7, CTR=0.15, freshness_days=10]
Doc B features: [BM25=6.1, PageRank=0.9, CTR=0.31, freshness_days=400]

A pointwise linear model with learned weights
  [w_bm25=0.5, w_pr=2.0, w_ctr=3.0, w_fresh=-0.01]:

score(A) = 0.5×8.2 + 2.0×0.7 + 3.0×0.15 + (-0.01×10)  = 4.1+1.4+0.45-0.1 = 5.85
score(B) = 0.5×6.1 + 2.0×0.9 + 3.0×0.31 + (-0.01×400) = 3.05+1.8+0.93-4.0 = 1.78

→ Doc A wins. Even though Doc B has lower BM25 AND higher CTR historically,
  its staleness (400 days, heavily penalized by the learned negative weight)
  drags it down. This is exactly the kind of multi-signal tradeoff no
  hand-written formula could reliably capture — the weights themselves were
  LEARNED from labeled data, not chosen by a human.
```

---

## Why it works / why it fails

**Works:**
- Combines arbitrarily many heterogeneous signals correctly, with interaction effects, instead of a hand-picked linear combination
- Directly optimizes the metric you actually care about (NDCG) rather than a proxy
- Retrains automatically as user behavior data grows — no manual re-tuning

**Fails:**
- **Needs labeled data** — either human relevance judgments (expensive, slow) or click logs (noisy, biased toward what was already ranked highly — **position bias**: users click top results more just because they're on top, not necessarily because they're most relevant, which can corrupt the training signal in a feedback loop)
- Tree ensembles and neural rankers are far less interpretable than "the score is TF times log of a ratio" — debugging why a specific document ranked where it did is much harder
- Adds real inference latency and infra complexity (Chapter 13) — LTR models are usually a re-ranking stage applied only to the top-k candidates from a cheap first-pass retriever, precisely because running a full tree ensemble or neural model over the entire corpus per query would be too slow

## Interview Q&A

**Q1. Why is pairwise loss (RankNet) an improvement over pointwise loss, and what does it still get wrong?**

Pointwise loss scores each document independently and never explicitly compares documents for the same query, so it can get every score "close enough" in absolute terms while still producing the wrong relative order — which is all that matters for ranking. Pairwise loss fixes this by directly penalizing wrong relative orderings between document pairs. What it still gets wrong is treating every misordered pair equally regardless of position — swapping the top two results is penalized identically to swapping results 47 and 48, even though only the former meaningfully hurts the user experience.

**Q2. Explain in one sentence what LambdaRank changes about RankNet, and why that specific change matters.**

LambdaRank multiplies RankNet's pairwise gradient by `|ΔNDCG|`, the change in the ranking metric that would result from swapping that specific pair — this matters because it makes training explicitly position-aware, so the model is pushed hardest to fix errors near the top of the ranking (where users actually look) and barely bothers with errors buried deep in the list.

**Q3. Why can't you just directly optimize NDCG with gradient descent?**

NDCG depends on the *sorted rank* of documents, and sorting is a discontinuous, non-differentiable operation — you can't take a smooth gradient through a "sort" step. LambdaRank sidesteps this entirely by not deriving its gradient from a loss function at all; instead it directly *defines* the gradient (`λ`) as the RankNet gradient scaled by `|ΔNDCG|`, giving position-aware training dynamics without ever needing NDCG itself to be differentiable.

**Q4. Why does production LTR usually use gradient-boosted trees (LambdaMART) rather than a neural network?**

Ranking features are highly heterogeneous — some continuous (BM25 score), some binary (mobile-friendly), some skewed (PageRank, freshness in days) — with strong nonlinear interaction effects between them. Tree ensembles handle this kind of mixed, non-smooth feature space and interaction structure very naturally without needing careful feature normalization or architecture design, which is part of why LambdaMART became the long-standing production standard, though large-scale systems increasingly layer neural rankers on top of or alongside tree-based first stages.

**Q5. What is position bias in click-log training data, and why is it dangerous for LTR?**

Position bias is the tendency of users to click on top-ranked results more often simply because they are shown first and get more attention, independent of whether they're actually the most relevant result. If click logs are used as training labels without correcting for this, the model learns "things that were ranked highly before get more clicks" and reinforces the existing ranking rather than learning true relevance — a feedback loop that can entrench a mediocre ranking rather than improve it. Production systems correct for this with techniques like randomized result interleaving or explicit position-bias models.

# Chapter 13 — Scalability and Infrastructure

## What is it?

Every technique in Chapters 2–12 assumed, implicitly, that you could compute `score(q, d)` for every document `d` in the corpus, for every query, fast enough to serve a user in under a few hundred milliseconds. At web scale (billions of documents, thousands of queries per second), that assumption breaks completely. Chapter 13 is about the systems engineering that makes retrieval actually work at scale — this is where IR meets distributed systems.

---

## 13.1 Sharding strategies

A single machine cannot hold a billion-document inverted index in memory, and even if it could, one machine can't answer thousands of queries per second alone. The index gets **split (sharded)** across many machines.

### By document ID (the default choice)

Documents are assigned to shards essentially at random (e.g. `shard = hash(doc_id) % num_shards`), so each shard ends up with a random, roughly-equal-sized slice of the whole corpus.

```
Shard 0: docs {3, 17, 892, 1500, ...}
Shard 1: docs {8, 44, 901, 1502, ...}
Shard 2: docs {1, 19, 850, 1499, ...}
```

**Every query must go to every shard**, because any shard might contain the best-matching document (Chapter 13.2 — scatter-gather).

**Why this is the default:** load balancing is trivial and automatic — since assignment is randomized, each shard gets a statistically even share of both storage and query traffic, with no risk of "hot shards."

### By topic cluster

Documents are grouped by topic (e.g. via clustering or category metadata) and each shard holds one topic cluster.

```
Shard 0: all "sports" documents
Shard 1: all "finance" documents
Shard 2: all "cooking" documents
```

**Advantage:** if a query is confidently classified as topic-specific ("NBA finals score"), you can route it to *only* the sports shard, skipping the rest entirely — much cheaper per query.

**Why this is rarely the default:** real corpora and real queries are rarely cleanly single-topic. A query like "diet plans for marathon runners" straddles cooking/nutrition and sports. Misclassifying the query's topic means missing the best-matching document entirely — silently, with no error, just a wrong (or empty) result. It also creates **hot shards**: if "finance" documents get queried far more often than "cooking" documents, that shard becomes a bottleneck while others sit idle, breaking the load-balancing property that document-ID sharding gets for free.

### Comparison

| | By document ID | By topic cluster |
|---|---|---|
| Load balancing | Even, automatic | Uneven, needs manual rebalancing |
| Query cost | Always hits every shard | Can skip irrelevant shards (cheaper, if classification is right) |
| Failure mode | None specific to sharding | Misclassified query → missing the best document silently |
| Used in practice | **Yes, overwhelmingly** | Rare; sometimes combined as a secondary routing hint on top of doc-ID sharding |

---

## 13.2 The scatter-gather query pattern

Given document-ID sharding, a single query physically has to visit every shard. This 2-phase pattern is the standard way that happens:

```
                     ┌─────────────┐
   query q  ────────▶│  Broadcaster │
                     └──────┬──────┘
                            │ (scatter: send q to ALL shards in parallel)
              ┌─────────────┼─────────────┐
              ▼             ▼             ▼
         ┌────────┐   ┌────────┐    ┌────────┐
         │ Shard 0 │   │ Shard 1 │    │ Shard 2 │
         │ top-k=10│   │ top-k=10│    │ top-k=10│
         └────┬────┘   └────┬────┘    └────┬────┘
              │             │             │
              └─────────────┼─────────────┘
                            ▼ (gather: merge partial results)
                     ┌──────────────┐
                     │ Merge/re-rank │───▶ final top-10 to user
                     └──────────────┘
```

**Scatter phase:** the query is broadcast to every shard simultaneously (in parallel, not sequentially — this is critical for latency).

**Each shard independently** computes its own local top-k (e.g. top 10) using whatever scoring function is in play (BM25, then LTR re-ranking — Chapter 12).

**Gather phase:** a merging layer combines all shards' local top-k lists (e.g. 10 shards × top-10 each = 100 candidates) into one final global top-k by re-comparing scores across shards.

### The critical latency property

**Overall query latency ≈ latency of the SLOWEST shard, not the sum of all shards** — because the scatter step is parallel. This is why **tail latency** (p99, p999 — the 99th/99.9th percentile slowest response) matters far more in distributed search than average latency: one slow/overloaded/GC-pausing shard out of a thousand can single-handedly blow the latency budget for every query that touches it (which, under document-ID sharding, is *every* query).

### Why it works / why it fails

**Works:** parallelism means adding more shards mostly doesn't add latency (each shard only searches its own smaller slice), letting you scale corpus size roughly horizontally.

**Fails:**
- **Tail latency amplification.** With `N` shards all needing to respond, and each shard having some small independent probability of being briefly slow, the *chance that at least one shard is slow* grows with N. A query touching 1000 shards is far more likely to hit at least one slow straggler than a query touching 10 shards — this is why real systems set aggressive per-shard timeouts and simply return the fast shards' results, accepting a slightly incomplete answer over a slow, complete one.
- **Gather-phase cost.** Merging `N × k` candidates and re-ranking them centrally is itself work that grows with the number of shards; it doesn't get to skip this cost just because shard-level search was parallel.

---

## 13.3 Index replication for fault tolerance

Sharding solves *size* (the index doesn't fit on one machine). Replication solves *availability* (a single machine dying shouldn't take down a slice of the entire index).

```
Shard 0 → Replica A (primary), Replica B (backup), Replica C (backup)
Shard 1 → Replica A (primary), Replica B (backup), Replica C (backup)
```

Queries are load-balanced across a shard's replicas (any replica can answer, since they hold identical index data), and if the primary replica for a shard goes down, one of the backups is promoted, or simply continues serving reads, with no gap in the shard's availability.

**Tradeoff:** replication costs `N×` the storage and `N×` the indexing write cost (every document update has to propagate to every replica), in exchange for availability and read-throughput. This is a direct, classic instance of the **CAP theorem** tradeoff space (consistency vs. availability under partition) — search systems overwhelmingly bias toward availability and eventual consistency (it's fine if a replica is a few seconds stale on a brand-new document) rather than strict consistency, because a slightly-stale search result is a far smaller user-facing problem than a search engine returning an error.

---

## 13.4 Real-time vs. batch indexing (Kafka pipelines)

### Batch indexing

The classical approach: periodically (e.g. nightly) rebuild the entire inverted index from scratch across the whole corpus.

**Fine for:** corpora that don't change fast (archival document sets, a textbook corpus). **Bad for:** anything where freshness matters — a news query for something that happened 10 minutes ago must not wait for tonight's batch job.

### Real-time / streaming indexing

New/updated documents are indexed continuously as they arrive, typically via a message queue like **Kafka**:

```
New/updated doc ──▶ Kafka topic ──▶ Indexer consumers ──▶ update inverted index
                                     (parallel consumers,
                                      one per shard or partition)
```

Kafka's role specifically: it decouples document *ingestion* (which can spike unpredictably — a viral news event, a flash sale) from *indexing* (which has to keep up but shouldn't fall over under a spike). The queue absorbs bursts; indexer consumers drain it at a sustainable, controlled rate, and if an indexer crashes mid-stream, Kafka's offset-tracking means it resumes exactly where it left off rather than losing or duplicating updates.

### The realistic hybrid

Production systems essentially always run **both**: a batch pipeline occasionally rebuilds cleanly-optimized index segments from scratch (better compression, better term statistics), while a real-time streaming layer handles fresh documents in near-real-time, with periodic merges folding the real-time segments into the batch-optimized structure (conceptually similar to how Elasticsearch/Lucene segment merges work, referenced back in Chapter 2's Q5).

### Why it works / why it fails

**Works:** decouples "how fresh must results be" from "how efficient is the index," letting each side of the system be optimized independently.

**Fails:** streaming/real-time indexing introduces genuine **consistency complexity** — a document might be indexed on shard 3 but the update to a global statistic like IDF's `df(t)` or `N` (Chapter 2) hasn't propagated everywhere yet, meaning different shards can transiently disagree about term rarity for the same corpus. Most systems accept this as a bounded, self-correcting inconsistency rather than trying to eliminate it, because strict global consistency at this scale would be prohibitively slow.

## Interview Q&A

**Q1. Why does sharding by document ID remain the default over sharding by topic, even though topic sharding can skip irrelevant shards?**

Document-ID sharding gives automatic, even load balancing because assignment is essentially random — no shard becomes a hot spot. Topic sharding requires reliably classifying every incoming query's topic, and misclassification means silently missing the best-matching document with no visible error. It also creates uneven load, since some topics get queried far more than others. The efficiency gain from skipping shards is real but is outweighed, in most production systems, by the operational risk and the load-balancing loss.

**Q2. In a scatter-gather query touching 1,000 shards, why does adding more shards eventually hurt tail latency even though each individual shard is doing less work?**

Because the scatter step waits on every shard to respond (or times out), overall query latency is bounded by the *slowest* responding shard, not the average. As the number of shards grows, the probability that *at least one* of them is transiently slow (GC pause, disk hiccup, momentary overload) increases, even if each shard's individual chance of being slow stays constant. More shards means more chances for a straggler, which is why real systems impose aggressive per-shard timeouts and simply proceed with whichever shards responded in time.

**Q3. What specific problem does replication solve that sharding does not, and what's the direct cost?**

Sharding solves index *size* — splitting a corpus too large for one machine across many. It does nothing for *availability* — if a shard's one machine goes down, that entire slice of the index becomes unreachable. Replication solves this by keeping multiple identical copies of each shard, so a failed replica can be routed around. The direct cost is roughly linear: N replicas means N× the storage and N× the indexing write cost, since every update must propagate to every copy.

**Q4. Why is Kafka (or a similar message queue) specifically useful for real-time indexing, rather than writing directly to the index on ingestion?**

Document ingestion rates can spike unpredictably (breaking news, a flash sale), while the indexing pipeline needs to process updates at a steady, sustainable rate to avoid falling over. A message queue decouples these two rates: bursts of incoming documents are absorbed into the queue, and indexer consumers drain it at whatever pace keeps the index healthy. Kafka's offset-tracking also means a crashed indexer consumer resumes exactly where it left off, rather than silently losing or duplicating document updates.

**Q5. Real-time indexing means different shards can transiently compute different IDF values (`df(t)`, Chapter 2) for the same term. Why is this tolerated instead of fixed?**

Because enforcing strict global consistency for a corpus-wide statistic like `df(t)` across every shard, on every update, in real time, would require expensive cross-shard coordination on the hot write path — directly hurting ingestion throughput and latency. Since IDF drift of this kind is small and self-correcting (it converges once updates propagate, and periodic batch/segment-merge jobs re-normalize it — Chapter 2, Q5), production systems accept it as a bounded, temporary inconsistency rather than pay the cost of eliminating it entirely. This is the same availability-over-consistency tradeoff that motivates replication (13.3).

# Chapter 14 — Specialized IR

## What is it?

Chapters 2–13 built one general-purpose retrieval stack: text query in, text document out, scored and ranked at scale. Chapter 14 looks at four cases where that general stack needs to be specifically adapted because the query, the document, or the *conversation itself* isn't a simple flat piece of text.

---

## 14.1 Multimodal retrieval (image + text, CLIP-style)

### The problem

"Find me pictures of a dog playing in the snow" — the query is text, but the documents are images. TF-IDF, BM25, even standard dense retrieval (Chapter 4) operate on text-to-text similarity; none of them have any native way to compare a text query against a pixel grid.

### The idea — shared embedding space

**CLIP** (Contrastive Language-Image Pretraining) trains two separate encoders — one for images, one for text — such that a matching (image, caption) pair end up **close together in the same shared embedding space**, and non-matching pairs end up far apart.

```
"a dog playing in the snow" ──▶ [text encoder] ──▶  vector (512-dim)
     [photo of a dog in snow] ──▶ [image encoder] ──▶  vector (512-dim)

                    if they're a true match: vectors are close (high cosine similarity)
                    if mismatched: vectors are far apart
```

### Training mechanism (contrastive learning, briefly)

CLIP trains on a large batch of (image, caption) pairs. For each image, its *true* caption should score highest cosine similarity among *all* captions in the batch, and vice versa for each caption against all images. The loss pushes true pairs together and every other (image, wrong-caption) combination apart, simultaneously, across the whole batch.

### Why it works / why it fails

**Works:** once trained, retrieval becomes ordinary nearest-neighbor search (Chapter 4's approximate nearest neighbor machinery applies directly) — you embed the text query once, then find the nearest image vectors. No manual tagging of images required at all; CLIP learns visual-semantic association directly from naturally-occurring (image, caption) pairs at internet scale.

**Fails:**
- **Compositional reasoning is weak.** CLIP-style models are notoriously bad at binding attributes correctly — "a red cube on top of a blue sphere" vs. "a blue cube on top of a red sphere" can embed suspiciously close together, because the model learned a bag-of-concepts association more than true compositional structure (echoing TF-IDF's own "no word order" weakness from Chapter 2, in a different modality).
- Inherits whatever biases exist in the internet-scraped (image, caption) training pairs, since there's no explicit relevance labeling step to correct for it.

---

## 14.2 Code search (AST-aware tokenization, tree-sitter)

### The problem

A developer searches for `"function that reverses a linked list"` or pastes a code snippet and asks "find similar code." Code is text, technically — but treating it as plain natural-language text throws away almost everything that makes code *code*.

### Why naive tokenization fails on code

```
def reverse_list(head):
    prev = None
    while head:
        nxt = head.next
        head.next = prev
        prev = head
        head = nxt
    return prev
```

Plain whitespace/word tokenization (Chapter 2's `count(t in d)`) treats `head`, `nxt`, `prev` as arbitrary strings no different from natural-language stopwords, and has zero notion that this function *structurally* is a loop that mutates pointers — a renamed version of the exact same function (`def flip(node): temp=None; ...`) would share almost no tokens at all despite being semantically identical.

### The fix — parse into an AST, tokenize structure

Tools like **tree-sitter** parse source code into an **Abstract Syntax Tree (AST)** — a structural representation of the code (function definitions, loops, conditionals, variable assignments) independent of variable naming.

```
source code ──▶ [tree-sitter parser] ──▶ AST:
                                           FunctionDef
                                           ├── Parameter (head)
                                           ├── Assign (prev = None)
                                           ├── While
                                           │   ├── Assign (nxt = head.next)
                                           │   ├── Assign (head.next = prev)
                                           │   ├── Assign (prev = head)
                                           │   └── Assign (head = nxt)
                                           └── Return (prev)
```

Retrieval features are then extracted from this **structural** representation — e.g. "contains a while-loop that reassigns a pointer 3 times" — rather than from surface tokens, or the AST itself is fed into a code-specific embedding model (e.g. CodeBERT-style) so that semantically identical, differently-named functions land near each other in embedding space, the same shared-space idea as CLIP (14.1) but for code structure instead of images.

### Why it works / why it fails

**Works:** structurally identical code with different variable names, different comment style, or reordered-but-equivalent statements can still be found as "similar," which plain-text token search fundamentally cannot do.

**Fails:** AST parsing is language-specific (a tree-sitter grammar has to exist for each language you want to support) and brittle to syntax errors — a snippet that doesn't fully parse can't be AST-tokenized at all, forcing a fallback to plain-text search for broken/partial code, which is common in real developer search scenarios (searching mid-edit, searching error logs containing code fragments).

## 14.3 Conversational search (multi-turn queries, context carryover)

### The problem

```
Turn 1 — User: "What's the population of Japan?"
Turn 2 — User: "What about South Korea?"
```

Turn 2, read in isolation, is not even a well-formed retrieval query — "what about South Korea?" about *what*? A stateless retrieval system (everything in Chapters 2–13) treats every query independently and has no way to know Turn 2 is implicitly asking about population.

### The idea — query rewriting using conversation history

Before retrieval runs, an intermediate step rewrites the current turn into a **self-contained, context-resolved query** using the preceding conversation:

```
raw turn 2 query:        "What about South Korea?"
conversation history:    ["What's the population of Japan?"]
rewritten query:         "What's the population of South Korea?"
```

This rewritten query is then handed to the *exact same* retrieval stack from every previous chapter — conversational search isn't a different retrieval algorithm, it's a **pre-processing layer that turns multi-turn context into a single well-formed query** the existing machinery already knows how to handle. (Query rewriting here is a conceptual cousin of query expansion, Chapter 11 — both transform the query before scoring — but rewriting resolves missing context/coreference, while expansion adds synonyms/related terms to an already-complete query.)

### Why it works / why it fails

**Works:** cleanly reuses the entire existing retrieval stack — no need to build a parallel "conversational" scoring function from scratch; all the ranking machinery (BM25, LTR, etc.) stays exactly the same downstream.

**Fails:**
- **Coreference and ellipsis resolution is genuinely hard** and gets harder the further back the needed context lives — "what about South Korea?" three turns after the original topic, with two unrelated turns in between, requires the rewriting step to correctly identify *which* earlier turn is still relevant, not just "the most recent one."
- **Topic shifts break naive carryover.** If turn 3 is "how's the weather in Seoul," a rewriting model that overly aggressively carries forward "population" context could wrongly rewrite it as "what's the population and weather in Seoul," polluting the query — a direct conversational-domain analog of query drift from PRF (Chapter 11.2).

## 14.4 Personalized search (user history as context)

### The problem

Two different users type the identical query **"jaguar."** User A has spent the last month browsing car-buying sites. User B has spent the last month browsing wildlife documentaries. A single global ranking, identical for both users, cannot be right for both.

### The idea

Add the user's own history as additional input/features to the ranking stage — this plugs in almost exactly at the **Learning to Rank feature vector** from Chapter 12:

```
feature vector for (query="jaguar", doc=D, user=U):
  [ BM25 score,
    PageRank,
    global CTR,
    ... ,
    user_U_click_history_similarity(D),     ← personalization signal
    user_U_category_affinity(D.category) ]  ← personalization signal
```

The LTR model (Chapter 12) already knows how to combine dozens of heterogeneous features with learned weights and interaction effects — personalization signals are simply more features fed into the exact same machinery, not a separate system.

### Why it works / why it fails

**Works:** the same query genuinely means different things to different users, and personalization is the only technique in this book that can resolve that — no amount of better text matching, expansion, or reranking on the query/document alone can distinguish User A's "jaguar" from User B's, because the disambiguating signal isn't in the query text at all, it's in the user.

**Fails:**
- **Cold start** — a brand-new user has no history, so personalization features are simply missing/zero, and the system degrades to the same global ranking everyone else gets until enough history accumulates.
- **Filter bubble risk.** Aggressive personalization toward a user's established interests can systematically suppress results outside their existing pattern (User A never sees the wildlife-jaguar result, even when it's genuinely what they wanted this one time), trading relevance-on-average for reduced diversity/serendipity of results.
- Requires storing and correctly handling per-user history at genuinely enormous scale — a privacy and infrastructure cost far beyond the corpus-level statistics (Chapter 2's `df(t)`, `N`) every prior chapter needed.

## Interview Q&A

**Q1. Why can't standard dense retrieval (Chapter 4) be used as-is for text-to-image search, and what does CLIP do differently?**

Standard dense retrieval trains a single encoder (or two encoders of the same modality) so that semantically similar *text* passages land near each other in embedding space — it has no mechanism for relating text to an entirely different modality like pixels. CLIP trains two separate encoders, one per modality, with a contrastive objective that specifically pulls matching (image, caption) pairs together and pushes mismatched pairs apart across large batches, producing one shared embedding space that both text queries and images can be projected into for direct nearest-neighbor comparison.

**Q2. Why does renaming variables break plain-text code search but not AST-based code search?**

Plain-text tokenization treats variable names as ordinary tokens, so `def reverse_list(head): ...` and a renamed `def flip(node): ...` share almost no surface tokens despite being functionally identical, causing exact-match retrieval to miss the connection entirely. AST-based approaches parse the code into its structural representation first (function definitions, loops, assignments) independent of identifier names, so the two functions can be recognized as structurally/semantically equivalent regardless of what the variables are called.

**Q3. In conversational search, why is query rewriting usually implemented as a separate step before retrieval rather than modifying the retrieval scorer itself?**

Because the ambiguity in a follow-up turn ("what about South Korea?") is a *language-understanding* problem — figuring out what the incomplete query actually means given prior context — not a *relevance-scoring* problem. Once the query is rewritten into a complete, self-contained form ("what's the population of South Korea?"), every retrieval technique from earlier chapters (BM25, dense retrieval, LTR) already knows exactly how to handle it unmodified. Keeping the concerns separate means the existing, well-tested retrieval stack doesn't need to be redesigned for conversational input.

**Q4. What's the direct connection between personalized search and Learning to Rank from Chapter 12?**

Personalization is implemented by adding user-specific signals (click history similarity, category affinity, past query patterns) as additional features in the same feature vector the LTR model already combines — BM25 score, PageRank, freshness, and so on. It isn't a separate ranking system; it's the same learned model, given more (and more personal) inputs. This is why the chapter frames personalization as a natural extension of Chapter 12 rather than a standalone technique.

**Q5. What is a filter bubble in the context of personalized search, and why is it a genuine cost, not just a hypothetical concern?**

A filter bubble happens when personalization signals systematically push results toward a user's already-established interests and away from anything outside that pattern — so a user who has only ever clicked on car content for "jaguar" queries keeps getting car results even in the rare case they actually meant the animal. It's a genuine cost because it trades away result diversity and serendipity for higher *average* click-through, meaning the system can look like it's performing better by its own optimization metric (CTR) while actually serving a narrower, less useful range of results to real users over time.
