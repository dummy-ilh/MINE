## 1. The Core Distinction

These two terms are often confused or used interchangeably. They are **not the same thing**.

```
┌─────────────────────────────────────────────────────────────────────┐
│  FEATURE IMPORTANCE                                                 │
│  "Which features matter most?"                                      │
│  → A scalar score per feature                                       │
│  → Global, model-level summary                                      │
│  → Answers: ranking / selection                                     │
│                                                                     │
│  EXPLAINABILITY                                                     │
│  "Why did the model make THIS prediction?"                          │
│  → A decomposition of a prediction                                  │
│  → Can be local (one sample) or global (whole model)                │
│  → Answers: reasoning / debugging / trust                           │
└─────────────────────────────────────────────────────────────────────┘
```

### Concrete Analogy

Imagine a doctor diagnosing patients.

- **Feature Importance** is like asking: *"Across all the patients you've ever seen, which symptoms did you find most useful overall?"* → The doctor says: "Blood pressure, age, and cholesterol matter most."

- **Explainability** is like asking: *"Why did you diagnose THIS specific patient with heart disease?"* → The doctor says: "This patient's blood pressure of 180 pushed the risk up sharply; their age of 35 pulled it down slightly; the final score crossed the threshold."

One is a **global property of the model**. The other is a **local property of a single prediction**.

### Why This Matters

| Use Case | What You Need |
|---|---|
| Feature selection | Importance |
| Debugging a wrong prediction | Local Explainability |
| Understanding model behaviour overall | Global Explainability |
| Regulatory compliance (GDPR "right to explanation") | Local Explainability |
| Model comparison | Importance |
| Building stakeholder trust | Both |
| Detecting data leakage | Importance |
| Identifying fairness issues (per-group) | Local Explainability |

## 2. The Taxonomy — Full Map

```
ML Interpretability
│
├── Feature Importance  ─────────────── "How much does each feature matter?"
│   ├── Model-Specific
│   │   ├── Impurity / Gini importance    (tree models)
│   │   ├── Coefficient magnitude         (linear models)
│   │   └── Attention weights             (transformers)
│   └── Model-Agnostic
│       ├── Permutation Importance
│       ├── Drop-Column Importance
│       └── SHAP global (mean |SHAP|)
│
└── Explainability  ──────────────────── "Why this prediction?"
    │
    ├── Scope: GLOBAL  ──── Whole model behaviour
    │   ├── Partial Dependence Plots (PDP)
    │   ├── Accumulated Local Effects (ALE)
    │   ├── Global Surrogate Models
    │   └── SHAP Summary / Beeswarm Plots
    │
    └── Scope: LOCAL  ───── Single prediction
        ├── LIME
        ├── SHAP (individual force plots)
        ├── Counterfactual Explanations
        └── Anchors (rule-based local explanation)
```

And cutting across both scopes:

```
Model-Agnostic  →  Works on ANY model (black box)
Model-Specific  →  Exploits model internals (faster but less general)
```

## 3. Feature Importance — Deep Dive

### 3.1 Impurity-Based Importance (Gini / Entropy)

**What it is:** In tree-based models, every split reduces node impurity. The total impurity reduction attributed to each feature (summed across all splits, all trees) is its importance.

**Formula:**
```
Importance(j) = Σ_{t: split on j} [n_t / N] × [impurity(t) − p_L×impurity(t_L) − p_R×impurity(t_R)]
```
where `n_t` = samples at node t, `N` = total samples, `p_L`, `p_R` = fraction going left/right.

**Numerical Example:**

Suppose a single decision tree splits on `Age` at node 1 and `Income` at nodes 2 and 3:

```
Node 1 (root):  n=100, Gini=0.48,  splits on Age
  Node 2 (left):   n=60, Gini=0.35, splits on Income
  Node 3 (right):  n=40, Gini=0.20, splits on Income

Age impurity reduction    = (100/100)×0.48 − (60/100)×0.35 − (40/100)×0.20
                          = 0.48 − 0.21 − 0.08 = 0.19

Income impurity reduction (node 2) = ... = 0.12
Income impurity reduction (node 3) = ... = 0.07
Income total                        = 0.19

Importance(Age)    = 0.19  (normalised: 50%)
Importance(Income) = 0.19  (normalised: 50%)
```

**Key biases:**
- **High-cardinality bias**: Continuous or high-cardinality features get more split opportunities → inflated importance. A random ID column can rank first.
- **Training-set only**: Measured on training data → reflects memorisation, not generalisation.
- **No direction**: Tells you the feature matters; not whether high values are good or bad.

---

### 3.2 Permutation Importance

Covered in depth in the companion guide. Key summary:

- Shuffle one feature at a time on the **test set**
- Measure performance drop
- Model-agnostic, any metric, no retraining

**Key bias:** Underestimates correlated features (model uses substitute).

---

### 3.3 Drop-Column Importance

**What it is:** Retrain the model with each feature removed. Measure performance loss.

```
Importance(j) = Score(full model) − Score(model trained without j)
```

**Numerical Example:**

| Model | Features Used | Test R² |
|---|---|---|
| Full | size, rooms, age, zip | 0.88 |
| Drop size | rooms, age, zip | 0.71 |
| Drop rooms | size, age, zip | 0.82 |
| Drop age | size, rooms, zip | 0.87 |
| Drop zip | size, rooms, age | 0.79 |

| Feature | Importance (R² drop) |
|---|---|
| size | 0.88 − 0.71 = **0.17** |
| rooms | 0.88 − 0.82 = **0.06** |
| age | 0.88 − 0.87 = **0.01** |
| zip | 0.88 − 0.79 = **0.09** |

**Advantage over permutation:** Correctly handles correlated features. The model can't fall back on a correlated substitute because the substitute is also present — the model has to relearn without feature j.

**Disadvantage:** Requires **N+1 model trainings** (N = number of features). Expensive. Also, a retrained model may learn to compensate with other features in ways the original model didn't — the importance reflects "can the model survive without this feature" not "how much does the trained model rely on it."

---

### 3.4 Coefficient Magnitude (Linear/Logistic Regression)

**What it is:** In a linear model `y = β₀ + β₁x₁ + β₂x₂ + ...`, the absolute value of each `βⱼ` after standardising features gives importance.

**Critical requirement:** Features must be **standardised** (zero mean, unit variance) before fitting. Otherwise the coefficient magnitude reflects the scale of the feature, not its importance.

**Numerical Example:**

```
Raw (not standardised):
  y = 3.2 + 0.0001 × income + 5.4 × is_homeowner

Is is_homeowner more important? Not necessarily — income is in dollars (large scale)
compresses its coefficient. This is MEANINGLESS for importance.

Standardised:
  y = 3.2 + 1.8 × income_std + 0.6 × is_homeowner_std

Now: income is 3× more important than is_homeowner by coefficient magnitude.
```

**Limitations:**
- Linear models only
- Multicollinearity makes coefficients unstable (can flip signs)
- Regularisation (L1/L2) distorts magnitudes

---

### 3.5 VIF — Variance Inflation Factor

Not strictly an importance method, but essential context when computing any importance in the presence of multicollinearity.

**Formula:**
```
VIF(j) = 1 / (1 − R²_j)

where R²_j = R² from regressing feature j on all other features
```

**Interpretation:**
- VIF = 1 → no correlation with other features
- VIF = 5 → moderate multicollinearity (importance estimates unreliable)
- VIF > 10 → severe multicollinearity (importance estimates untrustworthy)

**Numerical Example:**

| Feature | R² (regressed on others) | VIF |
|---|---|---|
| size_sqft | 0.95 | 20.0 → severe multicollinearity |
| num_rooms | 0.93 | 14.3 → severe |
| age | 0.10 | 1.11 → fine |
| zip | 0.05 | 1.05 → fine |

Permutation importance on size_sqft and num_rooms will be biased. Use grouped permutation or drop-column importance for those two.

## 4. Explainability — Deep Dive

### 4.1 Global vs Local Explainability

```
GLOBAL Explainability
  ↳ Summarises model behaviour ACROSS ALL predictions
  ↳ "On average, how does the model respond to feature X?"
  ↳ Tools: PDP, ALE, global surrogate, SHAP summary plots
  ↳ Use: model audit, stakeholder communication, understanding learned patterns

LOCAL Explainability
  ↳ Explains ONE specific prediction
  ↳ "Why did the model score THIS customer as high risk?"
  ↳ Tools: LIME, SHAP force plots, counterfactuals, anchors
  ↳ Use: regulatory compliance, debugging errors, customer-facing explanations
```

**Key insight:** A global explanation is NOT simply the average of local explanations. The average of local SHAP values does equal global importance (by design of SHAP), but LIME local explanations cannot simply be averaged to get a meaningful global picture.

---

### 4.2 Model-Specific vs Model-Agnostic

```
Model-Specific (uses internals)
  ↳ Faster, exact, but tied to one architecture
  Examples:
    - TreeSHAP (exploits tree structure, O(TLD²) vs naive O(2^p))
    - Attention weights (transformers)
    - Integrated Gradients (neural networks, uses gradient paths)
    - DeepLIFT (neural networks)
    - Grad-CAM (CNNs, saliency maps)

Model-Agnostic (black-box, only uses predictions)
  ↳ Slower, approximate, but universal
  Examples:
    - Permutation Importance
    - LIME
    - KernelSHAP
    - PDP
    - ALE
    - Counterfactuals
    - Anchors
```

## 5. Global Explainability Methods

### 5.1 Partial Dependence Plots (PDP)

![PDP Example](https://scikit-learn.org/stable/_images/sphx_glr_plot_partial_dependence_002.png)

*Image: PDP showing marginal effect of a feature on predicted outcome — the curve shows how prediction changes as feature value changes, averaged over all other features.*

**What it does:** Shows the **marginal effect** of a feature on the predicted outcome, averaging out all other features.

**Algorithm:**
```
For feature j with candidate values v₁, v₂, ..., vₖ:
  For each value vₗ:
    1. Take the dataset X
    2. Replace column j with vₗ for ALL rows
    3. Get predictions: f(X with col_j = vₗ)
    4. PDP(vₗ) = mean of those predictions
Plot PDP(vₗ) vs vₗ
```

**Numerical Example:**

Dataset: 4 samples, predict house price from [size, age].

```
Original:
  size = [1000, 1500, 2000, 2500], age = [5, 10, 15, 20]

To compute PDP for size at value 1500:
  Set all rows to size=1500, keep age as-is:
  Row 1: size=1500, age=5   → model predicts $215k
  Row 2: size=1500, age=10  → model predicts $200k
  Row 3: size=1500, age=15  → model predicts $190k
  Row 4: size=1500, age=20  → model predicts $180k
  PDP(size=1500) = mean(215, 200, 190, 180) = $196k

Repeat for size=1000, 2000, 2500:
  PDP(size=1000) = $155k
  PDP(size=2000) = $240k
  PDP(size=2500) = $285k

Plot: as size increases from 1000 to 2500, expected price goes from $155k to $285k.
```

**What it reveals:** The shape of the relationship — linear, plateau, non-monotonic. You can see if high age hurts or helps price, and whether the effect is constant or changes.

**Limitation — The Extrapolation Problem:**
When you set all rows to `size=2500`, some rows will have `age=5` (a very new, very large house). If this combination is rare or impossible in reality, the model is extrapolating. PDP averages over ALL marginal values including unrealistic combinations.

**2D PDP (Interaction Plot):**
Vary two features simultaneously, keep others marginalised. The plot is a heatmap showing how the joint combination affects prediction. Used to detect interactions.

---

### 5.2 Accumulated Local Effects (ALE)

![ALE vs PDP](https://christophm.github.io/interpretable-ml-book/images/pdp-vs-ale.png)

*Image: PDP extrapolates into unrealistic regions (dotted area); ALE stays within the data distribution by computing local differences.*

**Why ALE exists:** PDP has the extrapolation problem. ALE fixes this by computing differences **locally within the actual distribution**.

**Algorithm:**
```
For feature j:
  1. Sort all samples by their value of feature j
  2. Divide into k equal-sized intervals (bins)
  3. For each bin [a, b]:
     For each sample i in this bin:
       lower_pred = f(X with x_ij replaced by a)
       upper_pred = f(X with x_ij replaced by b)
       local_effect_i = upper_pred - lower_pred
     bin_effect = mean(local_effect_i) / (b - a)
  4. ALE(v) = cumulative sum of bin effects up to v (centred)
```

**Key difference from PDP:**
- PDP computes absolute predictions at fixed feature values
- ALE computes **differences** (local slopes) within each bin, then integrates
- ALE never pushes samples into regions they don't naturally occupy

**When they differ:**
When features are correlated, PDP and ALE can give very different shapes. ALE is generally the more trustworthy picture of the true marginal effect.

**Numerical Example:**

Suppose `size` and `age` are correlated (large houses are new). PDP sets `size=2500` for even old (age=20) houses → unrealistic → biased average. ALE would only compare predictions at size=2499 vs 2500 for houses that actually have size near 2500 (which are also new) → stays realistic.

### 5.3 Global Surrogate Models

**What it is:** Train a simple, interpretable model (decision tree, linear regression) to **mimic** the predictions of a complex black-box model.

```
Step 1: Get predictions from your black-box model on the training set
        y_hat = black_box.predict(X)

Step 2: Train a simple model on (X, y_hat) instead of (X, y)
        surrogate = DecisionTree(max_depth=4).fit(X, y_hat)

Step 3: Interpret the surrogate as an approximation of the black box
```

**Metric:** R² between surrogate's predictions and black-box's predictions (not the true labels). High R² → surrogate is a good approximation.

**Limitation:** The surrogate is only as faithful as its R² with the black box. An R² of 0.75 means the surrogate explains 75% of the black box's behaviour — the remaining 25% is a blind spot, possibly the most complex and critical part.

## 6. Local Explainability Methods

### 6.1 LIME — Local Interpretable Model-Agnostic Explanations

![LIME Explanation](https://raw.githubusercontent.com/marcotcr/lime/master/doc/images/lime.png)

*Image: LIME fits a simple linear model (dashed line) locally around a single prediction (bold cross), in a neighbourhood defined by perturbed samples (dots, sized by similarity to the original).*

**Core idea:** Globally, a model may be very complex. But **locally, around any single prediction**, it can be approximated by a simple linear model.

**Algorithm:**
```
To explain prediction for sample x*:

1. Generate N perturbed samples around x*:
   - For tabular data: randomly set each feature to 0 (absent) or 1 (present)
     interpretable representation; then map back to original feature space
   - For text: randomly remove words
   - For images: randomly hide superpixels

2. Get black-box predictions for all N perturbed samples

3. Weight each perturbed sample by its similarity to x*:
   weight_i = exp(-distance(x_i, x*)² / σ²)
   (samples closer to x* get higher weight)

4. Fit a sparse linear model (e.g., Lasso) on the weighted samples:
   explanation = argmin_g [Σ weight_i × (f(x_i) - g(x_i))²] + complexity(g)

5. The coefficients of g are the explanation for x*
```

**Numerical Example:**

Predicting loan default. Sample x*: age=45, income=$60k, debt=$20k. Model output: 0.72 (high risk).

LIME perturbs around x* (100 samples), weights by distance, fits a linear model:

```
LIME explanation for this prediction:
  debt_ratio (+0.35)    ← biggest contributor to high risk
  income     (-0.18)    ← reduces risk
  age        (+0.05)    ← slight positive effect on risk
  
Intercept: 0.50 (baseline risk)
Linear approximation: 0.50 + 0.35 - 0.18 + 0.05 = 0.72 ✓
```

**What this tells you:** For THIS specific applicant, their debt ratio drove the high-risk score most, and their income partially offset it.

**LIME Pitfalls:**

1. **Instability**: Run LIME twice on the same sample → can get different explanations. The sampling and weighting is random.

2. **Neighbourhood definition is heuristic**: The `σ` parameter defines how large the "local" neighbourhood is. Too small → too few samples, noisy. Too large → the linear approximation is poor.

3. **Faithfulness ≠ 100%**: The local linear model is an approximation. Its R² with the black box in that neighbourhood tells you how faithful the explanation is.

4. **Tabular LIME is unreliable**: Text and image LIME have natural interpretable representations (words, superpixels). Tabular LIME's "turn features on/off" mapping is arbitrary and produces unstable results.

### 6.3 Counterfactual Explanations

**What it is:** Find the **minimum change** to an input that flips the model's prediction.

```
"You were denied a loan. If your income had been $5k higher, you would have been approved."
```

**Formal definition:**
```
Find x' such that:
  1. f(x') = desired_class   (prediction flips)
  2. distance(x, x') is minimised   (minimal change)
  3. x' is plausible (lies in the data distribution)
```

**Numerical Example:**

Original sample: age=25, income=38k, credit_score=580. Prediction: Denied (p=0.72 default risk).

Counterfactual: age=25, income=**45k**, credit_score=**620**. Prediction: Approved (p=0.38 default risk).

Another counterfactual: age=25, income=38k, credit_score=**650**. Prediction: Approved (p=0.41).

**Key properties:**

- **Actionable**: Tells the user exactly what to change (useful for customer-facing explanations).
- **No model internals needed**: Works purely from predictions — fully model-agnostic.
- **Non-unique**: Many counterfactuals can exist. DiCE (Diverse Counterfactual Explanations) generates multiple diverse ones.
- **Plausibility matters**: A counterfactual that says "change age from 25 to 65" is technically valid but not actionable. Good algorithms enforce that x' lies near the real data distribution.

**Contrastive Nature:** Counterfactuals are fundamentally **contrastive** explanations — they answer "why X and not Y?" which is how humans naturally explain decisions.

### 6.4 Anchors

**What it is:** Find a rule (set of conditions) that is sufficient to produce the same prediction, with high probability, regardless of the values of other features.

```
"This passenger is predicted to survive because: Sex=Female AND Pclass=1"
(Anchor precision = 0.97: 97% of passengers satisfying this rule are predicted to survive)
```

**Formal definition:**
```
Anchor A is a set of predicates (conditions on features) such that:
  P(f(x') = f(x) | A(x') = True) ≥ τ

where τ is a precision threshold (e.g., 0.95)
```

**How it differs from LIME:**

- LIME gives you **coefficients** (soft, linear, every feature contributes a bit)
- Anchors give you **rules** (hard, if-then, sufficient condition for the prediction)

LIME: "Income contributed +0.35 to the risk score."
Anchor: "IF debt_ratio > 0.4 AND credit_score < 600 THEN prediction = High Risk (95% of the time)."

**Coverage:** The fraction of the dataset where the anchor applies. High precision + high coverage = a globally useful rule. High precision + low coverage = a very specific local rule.

## 7. SHAP — Complete Reference

![SHAP Overview](https://raw.githubusercontent.com/slundberg/shap/master/docs/artwork/shap_header.png)

*Image: SHAP unifies feature importance (global) and individual explanations (local) under a single framework rooted in game theory.*

### 7.1 Shapley Values from Game Theory

SHAP is grounded in **cooperative game theory**. The question it answers:

> If features are "players" collaborating to produce a prediction, what is each player's fair contribution?

**The Shapley formula:**
```
φⱼ = Σ_{S ⊆ F\{j}} [ |S|!(|F|−|S|−1)! / |F|! ] × [f(S∪{j}) − f(S)]

where:
  F         = full set of features
  S         = subset of features NOT including j
  f(S)      = model prediction using only features in S
              (other features replaced by their expected value)
  f(S∪{j}) − f(S) = marginal contribution of j to subset S
```

In plain English: average the marginal contribution of feature j across **all possible orderings** in which features could be added. Weight each ordering equally.

**Axioms SHAP satisfies (unlike other methods):**

| Axiom | Meaning |
|---|---|
| Efficiency | All φⱼ sum to the difference between the prediction and the baseline: Σφⱼ = f(x) − E[f(x)] |
| Symmetry | Two features with identical contributions get equal Shapley values |
| Dummy | A feature that never changes predictions gets φⱼ = 0 |
| Linearity / Additivity | Shapley values of a sum of two models equal the sum of their Shapley values |

No other feature importance method satisfies all four simultaneously.

---

### 7.2 Numerical Example — Computing Shapley Values by Hand

**Setup:** Predict house price with 2 features: `size` and `location`. Baseline (average prediction) = $200k.

For one specific house (size=large, location=good), the model predicts **$280k**.

We need to explain why the prediction is $80k above baseline.

**All possible orderings of 2 features:**

| Order | Adding size first | Adding location first |
|---|---|---|
| size → location | f({size}) − f({}) | f({size,loc}) − f({size}) |
| location → size | f({size,loc}) − f({loc}) | f({loc}) − f({}) |

**Compute each:**

```
f({})              = baseline = $200k
f({size})          = $245k  (large size alone boosts by $45k)
f({location})      = $225k  (good location alone boosts by $25k)
f({size,location}) = $280k  (both together)
```

**Marginal contributions:**

```
Order 1 (size first):
  size marginal     = f({size}) − f({})              = 245 − 200 = $45k
  location marginal = f({size,loc}) − f({size})      = 280 − 245 = $35k

Order 2 (location first):
  location marginal = f({loc}) − f({})               = 225 − 200 = $25k
  size marginal     = f({size,loc}) − f({loc})        = 280 − 225 = $55k
```

**Shapley values (average over orderings):**

```
φ(size)     = (45 + 55) / 2 = $50k
φ(location) = (35 + 25) / 2 = $30k

Efficiency check: 50 + 30 = 80 = 280 − 200 ✓
```

**Interpretation:** This specific house is $80k above average. Of that, size accounts for $50k and location for $30k.

### 7.3 SHAP Variants

| Variant | Best For | How It Approximates Shapley |
|---|---|---|
| **KernelSHAP** | Any black-box model | Weighted linear regression over coalitions; model-agnostic but slow |
| **TreeSHAP** | Tree models (RF, XGBoost, LightGBM) | Exact, polynomial-time algorithm exploiting tree structure |
| **LinearSHAP** | Linear models | Closed-form solution using feature correlations |
| **DeepSHAP** | Neural networks | Backpropagation-based approximation using DeepLIFT connections |
| **GradientSHAP** | Neural networks | Combines integrated gradients with SHAP |

**TreeSHAP vs KernelSHAP:**

- TreeSHAP is **exact** (not an approximation) for tree models
- TreeSHAP runs in `O(TLD²)` where T=trees, L=leaves, D=depth — much faster than KernelSHAP
- KernelSHAP is `O(2^p)` naive (but uses sampling to make it tractable)
- Always use TreeSHAP for tree models

**Independence vs Interventional vs Conditional SHAP:**

This is a subtle but important distinction:

- **Marginal/Independent SHAP (default):** When a feature is "absent" from coalition S, replace it with a random draw from its marginal distribution P(X_j). Fast, but ignores correlations → can create unrealistic inputs when features are correlated.

- **Conditional SHAP:** Replace absent features with draws from P(X_j | X_rest). Stays in the data distribution → realistic inputs. But harder to compute and can attribute importance to correlated features differently.

- **Interventional SHAP:** Directly intervenes (do-calculus) to set features. Most appropriate for causal interpretation but requires causal graph knowledge.

For most practical use, the default (marginal) SHAP is fine. When features are highly correlated, be aware that SHAP attribution may split credit across correlated features.

### 7.4 SHAP Plots

**Force Plot (Local)**

Shows how each feature pushes the prediction above or below the baseline for one sample.

```
Baseline: 200k
                    ┌──────────┐  ┌─────┐
[← lower risk ──── │  size    │  │ age │ ─────── higher risk →]
                    │  +50k    │  │ -5k │
                    └──────────┘  └─────┘
                         ↕
                    location +30k
                         ↕
                    Final prediction: $275k
```

Red bars push prediction up (increase price); blue bars push it down.

**Summary / Beeswarm Plot (Global)**

![SHAP Beeswarm](https://raw.githubusercontent.com/slundberg/shap/master/docs/artwork/boston_waterfall.png)

Features sorted by mean |SHAP value| (top = most important). Each dot is one sample. Colour = feature value (red = high, blue = low). X position = SHAP value (right = pushes prediction up).

Lets you see: (1) which features are globally important, (2) the direction of their effect, (3) whether high or low values drive predictions.

**Dependence Plot**

Scatter plot of SHAP value for feature j (y-axis) vs feature value (x-axis). Reveals the functional form of the feature's effect and interaction effects (colour by a second feature to see interactions).

**Waterfall Plot (Local)**

Stacked bar starting from baseline, each feature adding or subtracting its SHAP value until reaching the final prediction. Cleaner visual than force plot for presentations.

## 8. Bias-Variance Implications Across Methods

### The Core Framework

Every explainability method makes an **approximation**. The quality of that approximation has its own bias-variance decomposition:

```
Approximation Error = Bias² + Variance + Irreducible Noise
```

| Method | Primary Bias Source | Primary Variance Source |
|---|---|---|
| Impurity Importance | High-cardinality features; training set only | Low (deterministic) |
| Permutation Importance | Correlated features; train set if overfit | `1/n_repeats` |
| Drop-Column | Retrained model compensates; model variance | Model training variance |
| PDP | Extrapolation over unrealistic inputs | Low (averages over all data) |
| ALE | Bin width choice (too coarse = miss detail) | Bin sample count |
| LIME | Neighbourhood definition; linear approximation | High (random sampling) |
| SHAP (Kernel) | Coalition sampling | `1/n_samples_coalitions` |
| SHAP (Tree) | None (exact) | None (deterministic) |
| Counterfactuals | Distance metric choice | Optimisation instability |
| Anchors | Precision threshold choice | Rule search randomness |

### LIME's Variance Problem in Detail

LIME is the highest-variance explainability method in common use.

Run LIME 5 times on the same sample, with the same model:

```
Run 1: debt_ratio=+0.35, income=-0.18, age=+0.05
Run 2: debt_ratio=+0.28, income=-0.22, age=+0.11
Run 3: debt_ratio=+0.41, income=-0.13, age=+0.03
Run 4: debt_ratio=+0.19, income=-0.30, age=+0.17
Run 5: debt_ratio=+0.33, income=-0.20, age=+0.08
```

The ranking is consistent (debt_ratio > income > age) but the magnitudes vary significantly. The variance in LIME explanations can be large enough to matter in practice.

**Fix:** Run LIME multiple times and average, or switch to SHAP (which is deterministic for tree models).

### Overfitting and Its Effect on Explanations

If your underlying model overfits:

- **Permutation importance on train set:** noise features look important (covered in companion guide)
- **SHAP values on train set:** SHAP will correctly explain what the overfit model is doing (high SHAP for noise features) — but this is the model's flaw, not SHAP's
- **PDP on overfit model:** will show erratic, non-smooth curves because the model has memorised specific data points
- **LIME on overfit model:** locally, the linear approximation might fit well, hiding that the global model is meaningless

**Key insight:** Explainability methods faithfully explain the model they're given. They don't tell you if the model itself is good. A perfectly explained bad model is still a bad model.

## 9. The Rashomon Problem

Named after Kurosawa's film where the same event is described differently by each witness.

**In ML:** Many different models can achieve the same (or nearly the same) test accuracy on a dataset. This set of equally-good models is called the **Rashomon set**.

**Why it matters for importance/explainability:**

Different models in the Rashomon set can attribute very different importances to the same features:

```
Dataset: predict income from [education, zip_code, age, gender]
Test accuracy: all models achieve ~82%

Model A (RF):  education=0.45, zip=0.30, age=0.20, gender=0.05
Model B (GBM): zip=0.40, education=0.35, age=0.20, gender=0.05
Model C (SVM): education=0.50, age=0.30, zip=0.15, gender=0.05
```

All three models are equally accurate, but their feature importance rankings differ. **Which one is "correct"?**

There is no single correct answer. Implications:

1. **Feature importance is model-dependent**, not data-dependent. You're measuring what this model uses, not what the data contains.

2. **Never use one model's importance to make strong causal claims** — another equally valid model might tell a different story.

3. **For high-stakes decisions** (credit, hiring, medical), you should examine importances across several models from the Rashomon set and report the range.

4. **Fisher et al. (2019)** formalised this into "Model Reliance" and suggested reporting the **minimum model reliance** across the Rashomon set as a more robust importance measure.

## 11. Numerical Case Study — Same Dataset, Every Method

**Dataset:** Predict whether a credit card transaction is fraud.

Features: `amount`, `hour_of_day`, `merchant_category`, `distance_from_home`, `is_foreign`.

Model: Random Forest, AUC = 0.94 on test set.

---

### Impurity Importance (from RF .feature_importances_)

```
amount              0.38
distance_from_home  0.29
merchant_category   0.18
hour_of_day         0.10
is_foreign          0.05
```

Limitation: `amount` is continuous with high cardinality → may be inflated. `is_foreign` is binary → only 2 split values → possibly underestimated.

---

### Permutation Importance (test set, n_repeats=20)

```
Feature               Mean AUC drop    Std
distance_from_home    0.082            0.009
amount                0.071            0.011
is_foreign            0.068            0.008
merchant_category     0.045            0.007
hour_of_day           0.012            0.006
```

Notable: `is_foreign` jumps from rank 5 (impurity) to rank 3 (permutation). The impurity method underestimated it because it's binary (few splits). Permutation correctly captures its predictive power.

---

### PDP — amount

```
amount ($)    PDP (predicted fraud probability)
0–50          0.04
50–200        0.08
200–500       0.15
500–1000      0.31
1000+         0.48
```

Monotonically increasing — higher amounts → higher fraud probability. Clear, actionable signal.

---

### ALE — amount

```
amount ($)    ALE effect
0–50          −0.06  (below average fraud rate)
50–200        −0.01
200–500       +0.05
500–1000      +0.18
1000+         +0.33
```

ALE centres at zero (average effect). Shape is similar to PDP but values differ slightly because ALE avoids extrapolating `amount` into unrealistic combinations with `distance_from_home`.

---

### SHAP (TreeSHAP) — Global Summary

```
Feature               Mean |SHAP|
distance_from_home    0.142
amount                0.118
is_foreign            0.105
merchant_category     0.071
hour_of_day           0.023
```

Consistent with permutation importance in ranking. Additionally, the beeswarm plot reveals:
- High `distance_from_home` (red dots) has high positive SHAP → increases fraud risk
- Low `amount` (blue dots) has negative SHAP → decreases fraud risk
- `is_foreign=True` strongly pushes predictions up

---

### SHAP — Local Example

Transaction: amount=$850, hour=2am, merchant=gas_station, distance=320km, is_foreign=True. Prediction: 0.87 (fraud).

```
Baseline (average fraud rate): 0.06

  is_foreign = True         +0.31  ████████████████████████████████
  distance   = 320km        +0.28  ████████████████████████████
  amount     = $850         +0.18  ██████████████████
  hour       = 2am          +0.09  █████████
  merchant   = gas_station  +0.05  █████

  Total SHAP:               +0.81
  Final prediction:         0.06 + 0.81 = 0.87 ✓
```

---

### LIME — Same Transaction

LIME samples around this point, weights by distance, fits linear model:

```
is_foreign = True    +0.29
distance = 320km     +0.25
amount = $850        +0.21
hour = 2am           +0.08
merchant = gas       +0.04

Predicted by LIME: 0.87 (matches ✓)
```

Very similar to SHAP here, but this is coincidental for this sample — LIME can give very different values on the next run.

---

### Counterfactual

"What's the minimum change to make this transaction NOT flagged as fraud (prediction < 0.3)?"

```
Original:      amount=850, is_foreign=True,  distance=320, hour=2am  → 0.87
Counterfactual: amount=850, is_foreign=False, distance=320, hour=2am  → 0.28

One change: is_foreign=False drops prediction below threshold.
```

This tells the fraud team: the `is_foreign` flag is the single most impactful factor for this specific transaction.

---

### Anchor

"What rule is sufficient to almost always predict fraud for cases like this?"

```
Anchor: IF distance > 200km AND is_foreign = True
  → Precision: 0.93 (93% of transactions satisfying this rule are predicted fraud)
  → Coverage: 0.04 (4% of all transactions match this rule)
```

Compact, human-readable rule. High precision, narrow coverage — a very specific fraud pattern.

## 12. Practical Decision Guide

### When to Use What

```
Your question                          → Method to use
────────────────────────────────────────────────────────
Which features should I keep/drop?    → Permutation Importance (test set)
Why was THIS prediction X?             → SHAP force plot or LIME
How does feature X affect predictions  → PDP (no corr) or ALE (with corr)
  on average?
What's the minimum change to flip      → Counterfactual Explanations
  a prediction?
I need a business rule for this group  → Anchors
Is my model fair across subgroups?     → SHAP per subgroup
Regulatory explanation needed          → Counterfactual or SHAP waterfall
I need to compare many features fast   → Impurity importance (fast, approximate)
My model is a neural network           → Integrated Gradients, GradShap, LIME
My model is a tree (RF/XGB/LGBM)       → TreeSHAP (exact, fast)
My model is linear                     → Coefficient magnitude (after scaling)
Features are highly correlated         → ALE, grouped permutation, drop-column
```

### Model-Specific Decision Tree

```
What model type?
│
├── Linear / Logistic Regression
│   → Coefficients (after standardising)
│   → SHAP LinearExplainer
│
├── Tree Models (RF, GBM, XGBoost, LightGBM)
│   → TreeSHAP (fast, exact)
│   → PDP / ALE for shape of effects
│   → Impurity importance (quick but biased)
│
├── Neural Networks
│   → Integrated Gradients
│   → GradientSHAP / DeepSHAP
│   → LIME (slow but works)
│
└── Any Other Black Box (SVM, k-NN, ensembles)
    → KernelSHAP
    → Permutation Importance
    → LIME
    → Counterfactuals
```

## 13. Common Pitfalls

| Pitfall | Detail | Fix |
|---|---|---|
| Using impurity importance for feature selection | Biased toward high-cardinality features; measured on training set | Use permutation importance on test set |
| Treating SHAP as causal | High SHAP value ≠ the feature causes the outcome | Add domain knowledge; use causal graphs |
| Running LIME once | LIME has high variance; one run may be misleading | Run 5–10 times, check consistency |
| Comparing importances across different models | Different baseline performances make raw numbers incomparable | Normalise by baseline metric |
| Ignoring correlated features in PDP | PDP extrapolates into unrealistic joint distributions | Use ALE instead when features are correlated |
| Using global explanations for individual decisions | A globally unimportant feature can be locally decisive | Use local methods (SHAP, LIME) for individual cases |
| Explaining an overfit model | Explanations are faithful to the model, not to reality | Fix the model first; explanations of bad models are misleading |
| Forgetting the Rashomon problem | Another equally accurate model might show different importances | Check importance consistency across multiple model types |
