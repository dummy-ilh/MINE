## Chapter 1: The Problem BERT Solves

### 1.1 Why Not Just Use Word2Vec?

Before BERT, embeddings like Word2Vec gave every word a **fixed vector** regardless of context.

**Example:**
```
"I went to the river bank."
"I deposited money at the bank."
```
Word2Vec gives *bank* the **same vector** in both sentences. This is wrong — meaning is contextual.

### 1.2 Why Not RNNs / LSTMs?

RNNs process left-to-right. To understand word at position *t*, they only have context from positions 0 to *t-1*.  
- They struggle with **long-range dependencies**  
- They are **sequential** — can't be parallelized  
- Gradient vanishing kills learning over long sequences

### 1.3 Why Not GPT (left-to-right Transformer)?

GPT reads left → right only (unidirectional). For tasks like:
- Named entity recognition  
- Question answering  
- Sentiment analysis  

...you need context from **both sides** of a word simultaneously.

### 1.4 What BERT Does Differently

BERT = **Bidirectional Encoder Representations from Transformers**

- Reads the **entire sentence at once** (not left-to-right, not right-to-left — all positions simultaneously)
- Produces **contextual embeddings** — the same word gets a different vector depending on its neighbors
- Is **pre-trained** on massive unlabeled text, then **fine-tuned** on specific tasks

**Key insight:** BERT is an *encoder only* model. It is great at understanding/representation, not generation.

### 1.5 What BERT Is NOT Good At

| Task | BERT? | Why |
|------|-------|-----|
| Text classification | ✅ | Use [CLS] token |
| NER | ✅ | Token-level output |
| Question answering (extractive) | ✅ | Span prediction |
| Text generation | ❌ | Not a decoder |
| Translation | ❌ | Needs encoder-decoder |
| Open-ended dialogue | ❌ | GPT-style needed |

## Chapter 2: Tokenization & Vocabulary

### 2.1 WordPiece Tokenization

BERT uses **WordPiece** — a subword tokenization scheme.

**Why subwords?**
- Full words → vocabulary explodes (millions of words, rare words unseen)
- Characters → sequences too long, loses meaning
- Subwords → sweet spot: handles rare/new words by breaking them up

**Example:**
```
"unbelievable" → ["un", "##believe", "##able"]
"playing"      → ["play", "##ing"]
"embeddings"   → ["em", "##bed", "##ding", "##s"]
```
The `##` prefix means "this piece is a continuation of the previous token."

### 2.2 Special Tokens

BERT adds two special tokens to every input:

```
[CLS] I love machine learning [SEP] It is fascinating [SEP]
```

- **[CLS]** — Classification token. Prepended to every sequence. Its final hidden state is used for sentence-level tasks.
- **[SEP]** — Separator. Marks end of sentence A and sentence B (for two-sentence tasks).
- **[MASK]** — Used during pre-training (Chapter 9).
- **[PAD]** — Padding to fill shorter sequences to a fixed length.

### 2.3 Numerical Example

Input: `"The cat sat"`

After WordPiece tokenization + special tokens:
```
Tokens:   [CLS]  The   cat   sat  [SEP]
Token IDs:  101   1996  4937  2938   102
Position:    0     1     2     3      4
Segment:     0     0     0     0      0   ← all sentence A
```

These integer IDs are the input to BERT's embedding layer.

## Chapter 3: Embeddings in BERT (The Three-Layer Input)

BERT doesn't just embed tokens. It sums **three separate embeddings** for every token.

```
Final Input Embedding = Token Embedding + Positional Embedding + Segment Embedding
```

### 3.1 Token Embeddings

A lookup table of shape `[vocab_size × hidden_dim]` = `[30,522 × 768]` for BERT-base.

Each token ID maps to a 768-dimensional vector learned during pre-training.

**Numerical example (simplified to d=4):**
```
Token ID 1996 ("The") → [0.21, -0.45, 0.88, 0.12]
Token ID 4937 ("cat") → [0.55, 0.31, -0.22, 0.67]
```

### 3.2 Positional Embeddings

Transformers have **no inherent sense of order** — attention sees all positions simultaneously. Without positional embeddings, "the cat sat" and "sat cat the" would produce identical representations.

**BERT's approach:** Learned positional embeddings (not the sinusoidal ones from the original Transformer paper).

A lookup table of shape `[max_seq_len × hidden_dim]` = `[512 × 768]`.

```
Position 0 → [0.01, 0.03, -0.02, 0.04]   ← [CLS]
Position 1 → [0.10, -0.05, 0.08, -0.01]  ← "The"
Position 2 → [0.15, 0.02, -0.09, 0.06]   ← "cat"
Position 3 → [0.20, 0.07, 0.11, -0.03]   ← "sat"
```

**What happens without positional embeddings?**  
The model has no way to distinguish "The cat ate the mouse" from "The mouse ate the cat." Word order becomes invisible.

### 3.3 Segment Embeddings

Used for **two-sentence tasks** (e.g., NSP, QA). Tells the model which sentence each token belongs to.

```
Sentence A tokens → Segment Embedding A = [0.0, 0.0, 0.0, 0.0]  (learned vector)
Sentence B tokens → Segment Embedding B = [0.1, -0.1, 0.2, 0.05] (different learned vector)
```

For single-sentence tasks, all tokens use Segment A embedding.

### 3.4 Summing All Three (Numerical Example, d=4)

For token "cat" at position 2 in sentence A:

```
Token Embedding(cat)    = [0.55,  0.31, -0.22,  0.67]
Positional Embedding(2) = [0.15,  0.02, -0.09,  0.06]
Segment Embedding(A)    = [0.00,  0.00,  0.00,  0.00]
                          ──────────────────────────────
Input to Transformer    = [0.70,  0.33, -0.31,  0.73]
```

This summed vector is what enters the first Transformer block.

## Chapter 4: Self-Attention (The Heart of BERT)

This is the most important chapter. Master this and everything else follows.

### 4.1 The Intuition

For every word, self-attention asks:  
*"How much should I look at every other word in the sentence when computing my own representation?"*

For the word "bank" in "river bank":  
→ High attention to "river"  
→ Low attention to "I", "went", "to", "the"

### 4.2 The Three Matrices: Q, K, V

Every token's input embedding is projected into three vectors:

- **Q (Query):** What am I looking for?
- **K (Key):** What do I offer to others?
- **V (Value):** What information do I carry if someone attends to me?

These projections use learned weight matrices:
```
Q = X · W_Q    shape: [seq_len × d_k]
K = X · W_K    shape: [seq_len × d_k]
V = X · W_V    shape: [seq_len × d_v]
```

In BERT-base: `d_model = 768`, `d_k = d_v = 64` (per head, since 768/12 heads = 64).

### 4.3 The Attention Formula

```
Attention(Q, K, V) = softmax( Q·Kᵀ / √d_k ) · V
```

**Step by step:**

**Step 1:** Compute raw attention scores (how much does each token attend to each other?)
```
Scores = Q · Kᵀ        shape: [seq_len × seq_len]
```

**Step 2:** Scale by √d_k (prevents vanishing gradients from dot products getting too large)
```
Scaled = Scores / √64 = Scores / 8
```

**Step 3:** Apply softmax row-wise (each row sums to 1 — attention weights)
```
Weights = softmax(Scaled)    shape: [seq_len × seq_len]
```

**Step 4:** Weighted sum of Value vectors
```
Output = Weights · V         shape: [seq_len × d_v]
```

### 4.4 Fully Worked Numerical Example

Let's use 3 tokens: ["the", "river", "bank"], d_k = 2 (tiny for clarity).

**Input embeddings X (3×2):**
```
X = [[1.0,  0.0],   ← "the"
     [0.0,  1.0],   ← "river"
     [1.0,  1.0]]   ← "bank"
```

**Learned weight matrices (2×2):**
```
W_Q = [[1, 0], [0, 1]]   (identity, for simplicity)
W_K = [[1, 0], [0, 1]]
W_V = [[1, 0], [0, 1]]
```

**Compute Q, K, V:**
```
Q = X·W_Q = X = [[1,0],[0,1],[1,1]]
K = X·W_K = X = [[1,0],[0,1],[1,1]]
V = X·W_V = X = [[1,0],[0,1],[1,1]]
```

**Step 1: Raw scores = Q·Kᵀ**
```
Q·Kᵀ = [[1,0],[0,1],[1,1]] · [[1,0,1],[0,1,1]]

= [[1·1+0·0,  1·0+0·1,  1·1+0·1],    "the"  attending to [the, river, bank]
   [0·1+1·0,  0·0+1·1,  0·1+1·1],    "river" attending to [the, river, bank]
   [1·1+1·0,  1·0+1·1,  1·1+1·1]]    "bank"  attending to [the, river, bank]

= [[1, 0, 1],
   [0, 1, 1],
   [1, 1, 2]]
```

**Step 2: Scale by √d_k = √2 ≈ 1.414**
```
Scaled = [[0.71, 0.00, 0.71],
          [0.00, 0.71, 0.71],
          [0.71, 0.71, 1.41]]
```

**Step 3: Softmax row-wise**
```
Row 1 softmax([0.71, 0.00, 0.71]):
  e^0.71=2.03, e^0.00=1.00, e^0.71=2.03  → sum=5.06
  → [0.40, 0.20, 0.40]   ← "the" attends equally to itself and "bank"

Row 2 softmax([0.00, 0.71, 0.71]):
  e^0.00=1.00, e^0.71=2.03, e^0.71=2.03  → sum=5.06
  → [0.20, 0.40, 0.40]   ← "river" attends equally to "river" and "bank"

Row 3 softmax([0.71, 0.71, 1.41]):
  e^0.71=2.03, e^0.71=2.03, e^1.41=4.10  → sum=8.16
  → [0.25, 0.25, 0.50]   ← "bank" attends MOST to itself
```

**Attention weight matrix:**
```
A = [[0.40, 0.20, 0.40],
     [0.20, 0.40, 0.40],
     [0.25, 0.25, 0.50]]
```

**Step 4: Output = A·V**
```
Output = A · [[1,0],[0,1],[1,1]]

"the"   = 0.40·[1,0] + 0.20·[0,1] + 0.40·[1,1] = [0.80, 0.60]
"river" = 0.20·[1,0] + 0.40·[0,1] + 0.40·[1,1] = [0.60, 0.80]
"bank"  = 0.25·[1,0] + 0.25·[0,1] + 0.50·[1,1] = [0.75, 0.75]
```

**Notice:** "bank" [0.75, 0.75] is a blend of all tokens (including "river"), giving it context. If the context were "bank robbery" instead, the weights would shift and "bank" would get a different output vector.

### 4.5 Why √d_k Scaling Matters

For large d_k (e.g., 64), dot products grow large → softmax saturates → gradients vanish.

**Without scaling (d_k=64):**
```
Scores could be ~30-50 → softmax outputs like [0.99, 0.005, 0.005]
→ Essentially one-hot attention → poor learning
```
**With √64=8 scaling:**
```
Scores brought back to small range → softer distributions → richer gradients
```

## Chapter 5: Multi-Head Attention

### 5.1 The Problem with Single-Head Attention

A single attention head learns one type of relationship. But sentences have many simultaneous relationships:
- Syntactic (subject-verb agreement)
- Semantic (word meaning)
- Co-reference ("it" → what noun?)
- Positional proximity

### 5.2 How Multi-Head Attention Works

Run `h` attention operations **in parallel**, each with its own Q, K, V projection matrices.

```
head_i = Attention(X·W_Q_i, X·W_K_i, X·W_V_i)

MultiHead(X) = Concat(head_1, ..., head_h) · W_O
```

**BERT-base:** h=12 heads, each of dimension 64. Output: 12×64 = 768 → projected back to 768 via W_O.

### 5.3 What Do Different Heads Learn?

Research has shown different heads specialize:
- Some heads track **syntactic dependencies** (verb → object)
- Some track **positional patterns** (attending to next/previous token)
- Some track **co-reference** (pronoun → antecedent)
- Some heads are **redundant** (which is why pruning works)

### 5.4 Why Does Concatenation + W_O Work?

Each head produces a 64-d subspace of meaning. Concatenating gives a 768-d vector that mixes all relationship types. W_O then linearly recombines these subspaces into the final representation.

**What happens without multiple heads?** One 768-d attention head would need to capture all relationship types simultaneously, creating competition and worse representations.

## Chapter 6: The Feed-Forward Layer (FFN)

### 6.1 Structure

After every multi-head attention block, there is a **position-wise feed-forward network**:

```
FFN(x) = max(0, x·W_1 + b_1) · W_2 + b_2
```

- `W_1`: `[768 × 3072]`  — expands to 4× the hidden size
- `W_2`: `[3072 × 768]`  — projects back down
- Activation: **GELU** (not ReLU — BERT uses GELU for smoother gradients)

### 6.2 Why Expand to 4× and Then Compress?

The expansion creates a high-dimensional "thinking space" where the model can separate and recombine features non-linearly, then compress back to the standard dimension.

Analogy: Attention figures out *which* tokens to combine. The FFN figures out *how* to transform that combined information into a richer representation.

### 6.3 Why GELU Instead of ReLU?

**ReLU:** Hard zero below 0 — kills gradient for negative inputs entirely.  
**GELU (Gaussian Error Linear Unit):** Smooth, probabilistic gating — small negative values are kept (with small weight), better gradient flow.

```
GELU(x) ≈ 0.5·x·(1 + tanh(√(2/π)·(x + 0.044715·x³)))
```

### 6.4 Numerical Example (d=4 → d=8 → d=4)

Input x = [0.70, 0.33, -0.31, 0.73]

W_1 projects to 8 dims (simplified): say output before activation = [1.2, -0.4, 0.8, 0.1, -1.1, 0.5, 0.3, -0.2]

After GELU ≈ [1.10, -0.15, 0.73, 0.05, -0.14, 0.35, 0.15, -0.08]

W_2 projects back to 4 dims: say output = [0.85, 0.40, -0.25, 0.80]

This is the FFN output, to be added to the residual.

## Chapter 7: Layer Normalization & Residual Connections

### 7.1 Residual Connections (Skip Connections)

After every sub-layer (attention OR FFN), the **input is added back** to the output:

```
output = LayerNorm(x + SubLayer(x))
```

**Why?**
- Prevents **vanishing gradients** in deep networks (gradient can flow directly through skip connection)
- Allows **identity mapping** — if a layer learns nothing useful, the residual keeps the signal intact
- Makes training stable with 12–24 layers

**What happens without residuals?**  
In deep networks, gradients shrink exponentially during backprop. Without skip connections, BERT's 12 layers would be nearly untrainable.

### 7.2 Layer Normalization

Normalizes across the **feature dimension** (not batch dimension like BatchNorm):

```
LayerNorm(x) = γ · (x - μ) / (σ + ε) + β
```
- μ, σ computed across the 768 features of a single token
- γ, β are learned scale and shift parameters

**Why LayerNorm instead of BatchNorm?**
- Sequences vary in length — batch statistics are unstable
- LayerNorm operates per-token, no dependence on batch size
- Works identically at training and inference

**Numerical Example:**

Token vector x = [0.70, 0.33, -0.31, 0.73] (d=4)
```
μ = (0.70 + 0.33 + (-0.31) + 0.73) / 4 = 1.45 / 4 = 0.3625
σ² = mean of (xi - μ)² = [(0.3375)² + (-0.0325)² + (-0.6725)² + (0.3675)²] / 4
   = [0.1139 + 0.0011 + 0.4523 + 0.1351] / 4 = 0.1756
σ = 0.419

Normalized: [(0.70-0.3625)/0.419, (0.33-0.3625)/0.419, (-0.31-0.3625)/0.419, (0.73-0.3625)/0.419]
          = [0.805, -0.078, -1.607, 0.877]
```
Then scaled and shifted by learned γ and β.

## Chapter 8: Stacking Transformer Blocks

### 8.1 One Transformer Block = Attention + FFN + Residuals

```
Block(x):
  a = LayerNorm(x + MultiHeadAttention(x))
  output = LayerNorm(a + FFN(a))
```

This block is repeated **N times**:
- BERT-base: N=12 blocks, 768 hidden, 12 heads → 110M parameters
- BERT-large: N=24 blocks, 1024 hidden, 16 heads → 340M parameters

### 8.2 What Each Layer Learns (Empirically)

| Layer Range | What's Learned |
|-------------|---------------|
| Layers 1-4 | Syntax, part-of-speech, low-level patterns |
| Layers 5-8 | Semantic relationships, co-reference |
| Layers 9-12 | Task-specific representations, high-level semantics |

This is why fine-tuning often only updates upper layers — lower layers encode general linguistic structure.

### 8.3 The Data Flow

```
Input Tokens
    ↓
[Token + Position + Segment Embeddings]  →  shape: [seq_len × 768]
    ↓
Transformer Block 1                      →  shape: [seq_len × 768]
    ↓
Transformer Block 2                      →  shape: [seq_len × 768]
    ↓
    ...
    ↓
Transformer Block 12                     →  shape: [seq_len × 768]
    ↓
Final Hidden States: one 768-d vector per input token
```

Each block refines the representation. By block 12, each token's 768-d vector is a rich, context-aware representation of that token in the context of the full sequence.

### 8.4 The [CLS] Token's Journey

[CLS] is a special token prepended to every input. It has no inherent meaning — but through 12 layers of attention, it **attends to all other tokens and aggregates information** from the entire sequence. By layer 12, its vector is used as the sentence-level representation for classification tasks.

## Chapter 9: BERT Pre-Training

### 9.1 Task 1: Masked Language Model (MLM)

**The problem:** For bidirectional training, you can't just predict the next word (that's GPT). Instead, BERT randomly masks 15% of tokens and predicts them.

**The 15% rule breaks down as:**
- 80% → replaced with [MASK]
- 10% → replaced with a random token
- 10% → kept as the original token

**Why not always [MASK]?**
- Fine-tuning never sees [MASK] tokens → train/test mismatch
- Random replacements force the model to maintain context even when the token isn't masked
- Keeping original forces the model to always produce good representations (not just for masked positions)

**Example:**
```
Original: "The cat sat on the mat"
Input:    "The [MASK] sat on the mat"    ← 80% case
Target:   predict "cat" at masked position
```

Loss: Cross-entropy on predicted token distribution vs. true token.

### 9.2 Task 2: Next Sentence Prediction (NSP)

Given two sentences A and B:
- 50%: B is the actual next sentence → label: IsNext
- 50%: B is a random sentence → label: NotNext

**Input:** [CLS] Sentence A [SEP] Sentence B [SEP]  
**Output:** Binary classification using [CLS] vector

**Why NSP?** Trains the model on sentence-level relationships, useful for tasks like QA and NLI (natural language inference).

**Note:** Later research (RoBERTa) showed NSP may not help much and can even hurt. RoBERTa removed NSP.

### 9.3 Training Scale

- Corpus: BooksCorpus (800M words) + English Wikipedia (2,500M words)
- BERT-base: 4 TPUs for 4 days
- BERT-large: 16 TPUs for 4 days
- Sequence length: 128 for 90% of steps, 512 for final 10%

## Chapter 10: Fine-Tuning BERT

### 10.1 The General Pattern

```
Pre-trained BERT weights
        ↓
Add a task-specific head (1-2 layers)
        ↓
Train on labeled task data (small)
        ↓
Task-specific BERT
```

Fine-tuning typically uses a **very small learning rate** (2e-5 to 5e-5) for 2-4 epochs.

### 10.2 Fine-Tuning for Different Tasks

**Text Classification (sentiment, topic):**
```
[CLS] this movie was great [SEP]
              ↓ (take [CLS] vector)
         Linear(768 → num_classes)
              ↓
         Softmax → class probabilities
```

**Named Entity Recognition (NER):**
```
[CLS] Obama visited Paris [SEP]
  ↓      ↓       ↓     ↓
(ignore) PER    ignore  LOC    ← per-token classification
```
Each token's 768-d vector → Linear(768 → num_entity_labels).

**Question Answering (SQuAD):**
```
Input: [CLS] Question [SEP] Context passage [SEP]
```
Task: find start and end positions of the answer span in the passage.

Two learned vectors (start_vec, end_vec) of shape [768]:
```
start_score_i = dot(start_vec, hidden_i)
end_score_i   = dot(end_vec,   hidden_i)
```
Softmax over all positions → highest scores give answer span.

### 10.3 What Parameters Update During Fine-Tuning?

Typically: **all BERT parameters + task head parameters**.  
In practice, lower layers change very little. This is why parameter-efficient fine-tuning (LoRA, adapters) works well.

## Chapter 11: BERT Variants

| Model | Key Difference | When to Use |
|-------|---------------|-------------|
| **BERT-base** | 12 layers, 110M params | Baseline, general use |
| **BERT-large** | 24 layers, 340M params | Max accuracy, lots of GPU |
| **RoBERTa** | No NSP, more data, longer training, dynamic masking | Better than BERT on most tasks |
| **DistilBERT** | 6 layers, 66M params, knowledge distillation | Speed-critical production |
| **ALBERT** | Parameter sharing across layers, factorized embeddings | Low memory, similar accuracy |
| **DeBERTa** | Disentangled attention (position & content separate) | State-of-the-art NLU |
| **SciBERT** | Pre-trained on scientific text | Science/medical NLP |
| **BioBERT** | Pre-trained on biomedical literature | Biomedical NLP |

### Why RoBERTa Outperforms BERT

1. Removed NSP (found to be unhelpful)
2. Dynamic masking (different tokens masked each epoch)
3. 10× more training data
4. Larger batch sizes
5. Longer training

## Chapter 12: Google Interview Patterns

### 12.1 Fundamental Theory Questions

**Q: Explain self-attention in one minute.**  
A: Each token computes a query, key, and value vector. Attention scores between all token pairs are computed as scaled dot products of Q and K, normalized with softmax, then used to weight-average the V vectors. Result: every token's representation is a context-sensitive blend of all tokens.

**Q: Why scale by √d_k?**  
A: Dot products in high dimensions grow large (variance ∝ d_k), pushing softmax into saturation (near-one-hot), which kills gradients. Scaling by √d_k brings variance back to ~1.

**Q: What's the complexity of self-attention?**  
A: O(n² · d) — quadratic in sequence length. This is BERT's main scaling limitation. (Compare: O(n · d²) for FFN.)

**Q: Why bidirectional? What can't it do?**  
A: Bidirectional = uses full context for each token. Can't do autoregressive generation (would attend to future tokens it needs to predict).

**Q: How is BERT different from GPT?**  
A: BERT = encoder only, bidirectional, for understanding. GPT = decoder only, unidirectional, for generation. GPT uses masked attention (can't see future).

### 12.2 Architecture Deep-Dives

**Q: What is the role of the FFN layer?**  
A: Attention captures *which* tokens relate. FFN transforms *how* each token's representation is computed — it's a per-token MLP that introduces non-linearity and increases model capacity.

**Q: Why residual connections in BERT?**  
A: Prevent vanishing gradients, allow identity mapping, enable effective optimization of 12-24 layer networks.

**Q: Why LayerNorm over BatchNorm?**  
A: NLP has variable sequence lengths, making batch statistics unstable. LayerNorm operates independently per token and is stable regardless of batch size or sequence length.

**Q: How does the [CLS] token work?**  
A: It's initialized randomly and has no inherent meaning. Through 12 layers of attention over the full sequence, it aggregates global information. Its final hidden state is used as the sequence-level representation for classification.

### 12.3 Training & Fine-Tuning Questions

**Q: What is the masking strategy in MLM and why not 100% [MASK]?**  
A: 80% [MASK], 10% random, 10% unchanged. If always [MASK], model never learns to represent real tokens and creates train/test mismatch. Random and unchanged tokens force robust representations at every position.

**Q: Why does BERT need so much pre-training data?**  
A: It has 110M-340M parameters and must learn general language structure (syntax, semantics, world knowledge) from scratch. The fine-tuning data is typically small, so the pre-trained weights carry most of the load.

**Q: What is catastrophic forgetting in BERT fine-tuning?**  
A: If fine-tuned with too high a learning rate, BERT forgets its pre-trained representations. Use small LR (2e-5 to 5e-5) and few epochs.

### 12.4 Practical / System Design Questions

**Q: You need to deploy BERT with 10ms latency. What do you do?**  
A: (1) Use DistilBERT (2× faster). (2) Quantize to INT8 (4× smaller, ~2× faster). (3) Use ONNX Runtime. (4) Trim sequence length. (5) TensorRT for GPU inference.

**Q: How would you handle sequences longer than 512 tokens?**  
A: (1) Sliding window with overlap, pool segment representations. (2) Use Longformer / BigBird (sparse attention, O(n) complexity). (3) Truncate intelligently (keep start + end of document for classification).

**Q: When would you NOT use BERT?**  
A: When you need generation (use GPT). When you need very low latency with very simple tasks (use TF-IDF + logistic regression). When data is highly domain-specific and no domain-adapted BERT exists.

### 12.5 Tradeoff Table

| Tradeoff | Choice A | Choice B |
|----------|----------|----------|
| Accuracy vs Speed | BERT-large | DistilBERT |
| General vs Domain | BERT | SciBERT/BioBERT |
| Long sequences | Standard BERT (512) | Longformer |
| Classification | [CLS] vector | Mean pool all tokens |
| Fine-tune all vs freeze | More data: fine-tune all | Less data: freeze lower layers |
