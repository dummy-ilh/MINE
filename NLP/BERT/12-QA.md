## 12.2 The Foundational Questions

---

### Q1: "Explain self-attention."

**Layer 1 answer (weak):**
"Self-attention lets each token look at other tokens."

**Layer 4 answer:**

```
"Self-attention computes three projections for every token —
query, key, and value — using learned weight matrices.

For each token, we compute dot products between its query
and every other token's key. This gives raw attention scores
measuring how relevant each token is to the current one.

We scale by √d_k to prevent dot products from growing too
large in high dimensions, which would saturate the softmax
and kill gradients.

After softmax, we have attention weights — a probability
distribution over all tokens. The output is a weighted
average of all value vectors using those weights.

The result: each token's representation becomes a
context-aware blend of all tokens in the sequence,
weighted by learned relevance.

The key insight is that relevance is not fixed — the query,
key, and value projections are all learned, so the model
discovers what 'relevant' means for each type of task."
```

---

### Q2: "Why scale by √d_k? What happens without it?"

**Weak:** "To normalize the scores."

**Strong:**

```
"In d_k dimensions, the dot product of two random unit vectors
has variance proportional to d_k. For d_k=64, scores can
easily reach magnitudes of 30-50.

When you feed large values into softmax, it produces near
one-hot distributions — one token gets weight ~1.0,
all others get ~0.0.

Two problems follow:

First, the model effectively attends to only one token
per head — losing the soft blending that makes attention
powerful.

Second, the gradient of softmax near saturation is nearly
zero — so the attention weights stop learning entirely.
Backprop through a saturated softmax gives almost no
gradient signal.

Dividing by √d_k brings variance back to ~1 regardless
of d_k, keeping softmax in a regime where gradients
flow and attention weights remain soft."
```

---

### Q3: "What is the complexity of self-attention and why does it matter?"

**Weak:** "It's O(n²)."

**Strong:**

```
"The score matrix is [seq_len × seq_len] — every token
attends to every other token. Computing this is O(n² · d_k)
in time and O(n²) in memory.

For n=512 (BERT's limit): 512² = 262,144 attention scores.
For n=1024:               1024² = 1,048,576 — 4× more.
For n=4096:               4096² = 16,777,216 — 64× more.

This quadratic scaling is why BERT caps at 512 tokens.
Going to 4096 would be 64× more memory for the attention
matrix alone — not feasible with standard full attention.

This is the core motivation for sparse attention models
like Longformer (local sliding window + global tokens,
O(n·w)) and BigBird (local + random + global, also O(n)).

In practice it also affects batching — long sequences
force smaller batch sizes, slowing training significantly."
```

---

### Q4: "Why is BERT bidirectional and why does that matter?"

**Weak:** "Because it reads the whole sentence at once."

**Strong:**

```
"BERT uses the encoder side of the Transformer, which has
no causal masking — every token can attend to every other
token in both directions simultaneously.

This matters because word meaning is determined by full
context, not just left context.

Classic example: 'bank' in 'river bank' vs 'savings bank'.
An RNN encoding left-to-right has seen only the words before
'bank' when it encodes 'bank.' If 'river' comes before and
'fish' comes after, the RNN captures 'river' but misses 'fish.'

BERT sees both simultaneously — 'river' and 'fish' both
contribute to 'bank's representation in the same layer.

The tradeoff: bidirectionality makes BERT unsuitable for
generation. To generate word t, you can't attend to
words t+1, t+2... because they don't exist yet.
That's why GPT uses unidirectional (causal) attention —
it masks out future tokens during attention.

BERT = encoder, understanding, bidirectional.
GPT  = decoder, generation, unidirectional.
This is the fundamental architectural fork."
```

---

### Q5: "Explain the MLM pre-training task. Why the 80/10/10 split?"

**Weak:** "BERT masks words and predicts them."

**Strong:**

```
"MLM randomly selects 15% of tokens and trains the model
to predict their original values using bidirectional context.

The 15% that are selected are treated three ways:
80% replaced with [MASK], 10% replaced with a random token,
10% kept unchanged.

If you always used [MASK], two problems arise:

First, [MASK] never appears during fine-tuning or inference.
The model develops a dependency on seeing [MASK] to know
it should predict something — creating a train/test
distribution mismatch that hurts downstream performance.

Second, the model learns to only care about building good
representations at masked positions. Non-masked positions
get no gradient from the MLM loss, so their representations
stagnate.

The 10% random token forces the model to distrust the
token it sees and rely heavily on context — building strong
representations everywhere.

The 10% unchanged forces the model to represent ALL tokens
well, since it can never know which positions will be
queried.

The result: BERT builds rich contextual representations
for every token in every position — not just masked ones.
This is what makes its embeddings so transferable."
```

---

### Q6: "What is the [CLS] token and why is it used for classification?"

**Weak:** "It's a special token that represents the whole sentence."

**Strong:**

```
"[CLS] is prepended to every input at position 0. It has
no inherent linguistic meaning — it's not a real word.

Its design purpose: through 12 layers of attention over
all other tokens, it becomes a vessel for global
sentence-level information.

Why position 0? Two reasons. First, it has no 'own meaning'
to preserve, so its representation is entirely shaped by
what it attends to. Second, the NSP pre-training task
specifically trained [CLS] to do sentence-level binary
classification (IsNext/NotNext) — it's already pre-trained
for the classification role.

By layer 12, [CLS] has attended to every token through
12 rounds of attention, with each round incorporating
progressively more abstract contextual information.
Its 768-d vector is a compressed summary of the full
input sequence.

The classification head — a single linear layer — maps
this 768-d vector to class logits.

Important nuance: [CLS] is not always the best choice.
For tasks with small fine-tuning datasets, mean-pooling
all token vectors sometimes outperforms [CLS] because
mean-pooling doesn't require [CLS] to calibrate its
classification behavior from few examples.
The right choice depends on the task and data size."
```

## 12.3 Architecture Deep-Dive Questions

---

### Q7: "What does the FFN layer do that attention can't?"

```
"Attention is fundamentally a weighted average of value
vectors — a linear operation over the sequence.

If you stacked 12 attention layers with no FFN between them,
the composition of linear operations is still linear.
No matter how deep, the model would have no more expressive
power than a single attention layer. Depth would be
meaningless.

The FFN introduces non-linearity through its GELU activation
between two linear projections. This makes each layer
genuinely more expressive than the previous one.

The FFN also serves a different function from attention:
attention figures out which tokens to gather information
from. The FFN figures out what to do with that gathered
information — it transforms each token's representation
independently and non-linearly.

Research suggests FFN layers also act as key-value memories,
storing factual knowledge learned during pre-training.
The expansion to 4× hidden size creates a high-dimensional
space where facts can be encoded and retrieved.
Factual recall tasks degrade more when FFN layers are
removed than when attention heads are removed."
```

---

### Q8: "Why residual connections? What breaks without them?"

```
"During backpropagation through 12 layers, the gradient
is multiplied by each layer's local gradient at every step.
If each local gradient has magnitude slightly less than 1
— say 0.8 — then by layer 1 the gradient is 0.8^12 ≈ 0.07.
7% of the original signal. Lower layers barely learn.

Residual connections add the input directly to the output:
output = x + F(x)

The gradient of this with respect to x is 1 + dF/dx.
The +1 term provides a direct gradient path regardless
of how small dF/dx becomes. Gradients can't vanish.

The second benefit: if a layer learns nothing useful,
F(x) → 0 and output → x. The layer becomes an identity
mapping, gracefully doing nothing instead of distorting
the signal.

Without residuals, training a 12-layer Transformer from
scratch is empirically nearly impossible. This was
demonstrated before ResNets — deep networks would plateau
or degrade as layers were added. ResNets, and then
Transformers, showed that residuals unlock depth."
```

---

### Q9: "Why LayerNorm instead of BatchNorm?"

```
"BatchNorm normalizes across the batch dimension — it
computes mean and variance across all samples in the
batch for each feature position.

Two problems in NLP:

First, sequences have variable lengths. Padding makes
batch statistics noisy and meaningless — you're averaging
real token representations with padding zeros.

Second, BatchNorm behaves differently at training vs
inference. At training it uses batch statistics; at
inference it uses running averages accumulated during
training. For NLP this discrepancy is problematic,
especially with small batch sizes where running averages
are poor estimates.

LayerNorm normalizes across the feature dimension —
computing mean and variance across the 768 dimensions
of a single token's vector.

This is completely independent of batch size and other
samples. It's identical at training and inference.
It works for any sequence length. And it stabilizes
the forward pass regardless of how large residual
additions make the activations.

The learned scale γ and shift β allow the model to
undo normalization if a layer benefits from different
statistics — so LayerNorm never forces a sub-optimal
distribution on the model."
```

## 12.4 Training and Fine-Tuning Questions

---

### Q10: "Why does fine-tuning use such a small learning rate?"

```
"Pre-trained BERT weights encode years of computation
on 3.3 billion words. They represent a rich initialization
in a high-dimensional weight space where language is
well-understood.

A large learning rate causes large gradient steps.
With 110M parameters, large steps can move the weights
far from their pre-trained values in a single epoch —
the model 'forgets' the pre-trained language knowledge.
This is catastrophic forgetting.

With a small learning rate (2e-5 to 5e-5), each step
makes tiny adjustments. The task-specific signal
accumulates gradually over 2-4 epochs, nudging the
upper layers toward task-relevant representations while
lower layers barely change.

Empirically: fine-tuning BERT-base with lr=1e-3
typically diverges or performs near-randomly.
With lr=2e-5, the same model reaches state-of-the-art
on most benchmarks.

The warmup schedule matters too — starting from lr=0
and warming up over 10% of steps prevents the randomly
initialized task head from sending large, destructive
gradient updates through the pre-trained layers in
the first few steps."
```

---

### Q11: "BERT vs GPT — when would you choose each?"

```
"Fundamental architectural difference:
BERT is encoder-only with bidirectional attention.
GPT is decoder-only with causal (left-to-right) attention.

Choose BERT when:
  The task is understanding/representation — you need
  to read and comprehend text, not generate it.
  Examples: classification, NER, extractive QA,
  sentence similarity, information retrieval.
  BERT sees full context for every token — better
  representations for understanding tasks.

Choose GPT when:
  The task requires generation — producing new text
  token by token. Examples: summarization, translation,
  dialogue, code generation, creative writing.
  GPT's causal attention is required for coherent
  autoregressive generation.

The grey zone — tasks like abstractive QA, instruction
following, chain-of-thought reasoning — GPT-style
models have largely won here because generation
flexibility matters more than bidirectionality.

In 2024 practice: for pure classification/extraction,
fine-tuned BERT-style models are still competitive
and much cheaper than large GPT models. For anything
requiring generation or instruction following, GPT-style
models dominate."
```

## 12.5 System Design Questions

These are the most differentiating questions. Most candidates know the theory. Fewer can design systems.

---

### Q12: "Design a production BERT system with 50ms latency budget."

Structure your answer with this framework:

```
Step 1: Understand the constraint
─────────────────────────────────
50ms total budget. BERT-base on CPU: ~100ms per sequence.
Already over budget. Need to optimize.

Step 2: Model optimization
──────────────────────────
Option A: Switch to DistilBERT
  → 6 layers, 66M params, 1.6× faster
  → ~60ms on CPU → still tight

Option B: Quantization (INT8)
  → Convert float32 weights to int8
  → 4× smaller model, ~2× faster inference
  → <1% accuracy drop on most tasks
  → BERT-base quantized: ~50ms on CPU

Option C: ONNX Runtime
  → Export model to ONNX format
  → ONNX Runtime optimizes graph execution
  → Operator fusion, memory layout optimization
  → 20-40% additional speedup

Combined: DistilBERT + INT8 quantization + ONNX
  → ~25-30ms per sequence on CPU ✓

Step 3: Input optimization
───────────────────────────
Most sequences don't need 512 tokens.
Analyze your data distribution:
  → If 95% of inputs < 64 tokens, cap at 64
  → Attention is O(n²) — 64 tokens is 64× faster than 512
  → Dynamic sequence length padding per batch

Step 4: Infrastructure
───────────────────────
Batching: group requests into batches of 32
  → GPU utilization goes from 5% to 80%
  → Throughput increases dramatically
  → Add max_wait_time=5ms to prevent long queue buildup

Caching: identical inputs get identical outputs
  → Cache (input_hash → output) in Redis
  → Repeat queries (common in production) hit cache

GPU vs CPU:
  → GPU: lower latency for large batches (1-5ms)
  → CPU: better for low-traffic, cost-sensitive

Step 5: Monitoring
───────────────────
Track: p50, p95, p99 latency
Track: model accuracy on held-out set weekly
Alert: latency > 45ms (buffer before SLA breach)
Alert: accuracy drop > 1% (model drift)
```

### Q13: "How would you handle documents longer than 512 tokens?"

```
"Four approaches, each with different tradeoffs:

Approach 1: Truncation
  Simply cut at 512 tokens.
  Works when: answer/label is in first 512 tokens
  (common for short-answer QA where answer is near start)
  Fails when: relevant content is later in the document

Approach 2: Sliding Window
  Split document into overlapping chunks of 512 tokens.
  Run BERT on each chunk independently.
  Aggregate chunk outputs (max pool, mean pool, or
  weighted by [CLS] confidence scores).
  Works well for: classification, where any chunk
  might contain the label signal
  Fails for: tasks requiring cross-chunk reasoning

Approach 3: Hierarchical BERT
  First BERT: encode each 512-token chunk independently
  → produces one [CLS] vector per chunk
  Second BERT (or Transformer): encode the sequence
  of [CLS] vectors as a 'sentence-level' sequence
  → produces document-level representation
  Works for: long document classification
  Requires: two-stage training

Approach 4: Long-Document Models
  Longformer: sparse local + global attention, 4096 tokens
  BigBird: local + random + global attention, 4096 tokens
  Both drop the O(n²) constraint to O(n)
  Works for: tasks needing full document context
  Costs: longer training, more complex infrastructure

My recommendation:
  Start with truncation (simplest baseline).
  If accuracy suffers, move to sliding window.
  If cross-document reasoning is needed, use Longformer.
  Profile latency at each step before adding complexity."
```

### Q14: "Your BERT model accuracy dropped 5% in production. Debug it."

This tests systematic thinking under pressure.

```
"I'd work through four hypotheses in order of ease to check:

Hypothesis 1: Data drift
  Production input distribution shifted from training data.
  Check: compare token length distribution, vocabulary,
  and topic distribution between training set and
  recent production traffic.
  Fix: collect recent production examples, add to
  fine-tuning data, retrain.

Hypothesis 2: Preprocessing mismatch
  Tokenization, lowercasing, or truncation behaves
  differently in production vs training.
  Check: log raw inputs and tokenized outputs in
  production, compare to training preprocessing pipeline.
  Common bug: training lowercases, production doesn't
  (or vice versa for cased/uncased BERT mismatch)
  Fix: ensure identical preprocessing.

Hypothesis 3: Label distribution shift
  The proportion of classes changed in production.
  Example: trained on 50/50 positive/negative,
  production is 90/10 — model appears less accurate
  because it's evaluated on a different distribution.
  Check: compute per-class accuracy, not just overall.
  Fix: recalibrate decision thresholds, or collect
  representative production data.

Hypothesis 4: Model serving issue
  Wrong model version deployed. Weights corrupted.
  Batch size change causing different padding behavior.
  Check: run a fixed test set through production endpoint,
  compare to offline evaluation. Should be identical.
  Fix: version control all model artifacts, checksum
  weights on deployment.

My process: I'd add detailed logging immediately to
capture raw inputs and outputs, run the fixed test set
against production, and compare metrics by input segment
(length, domain, etc.) to narrow down where the drop
concentrates."
```

## 12.7 The Questions That Catch People Out

These are specifically designed to expose shallow understanding:

---

**"If attention is O(n²), why does BERT use 512 and not 256?"**

```
"Tradeoff between context richness and compute.

256 tokens: faster, but many real sentences + context
get truncated. Tasks like QA where answer could be
anywhere in a paragraph would suffer.

512 tokens: covers most paragraphs and short documents.
The quadratic cost at 512 is 4× that of 256 —
acceptable with modern hardware.

The real answer: 512 was empirically chosen as the
sweet spot for the pre-training corpus used (Wikipedia
+ Books). Most Wikipedia paragraphs fit within 512.
Longer would give diminishing returns on this corpus
while increasing compute significantly."
```

---

**"Can you use BERT for language generation?"**

```
"Not directly. BERT is an encoder with bidirectional
attention — it can see future tokens. If you tried to
generate token by token, each generated token would
attend to the not-yet-generated future tokens, which
don't exist. The architecture assumes the full sequence
is available.

You could generate with BERT using iterative masking:
mask all positions, predict them all at once, then
refine iteratively. This is how BERT-based generators
like BERT-gen work. But it's much slower and lower
quality than GPT-style autoregressive generation.

For generation, use an encoder-decoder (T5, BART)
or decoder-only (GPT) architecture. BERT's power
is in understanding, not generation."
```

---

**"Why does RoBERTa outperform BERT despite same architecture?"**

```
"Three main reasons:

First, removing NSP. NSP forced short segment pairs
(A+B must fit 512 tokens), limiting context to ~256
tokens per segment. Without NSP, RoBERTa trains on
full 512-token sequences — richer context per step.

Second, dynamic masking. BERT masks once before
training — the model sees the same mask pattern
40 times across 40 epochs. RoBERTa generates
fresh masks each epoch, seeing 40× more diverse
training signal.

Third, scale. RoBERTa trained on 160B words vs
BERT's 3.3B. With larger batches (8000 vs 256),
gradient estimates are more stable and training
is more efficient.

The lesson: BERT was massively undertrained.
Given the same training budget as RoBERTa,
the original BERT architecture matches or exceeds
RoBERTa. Architecture matters, but so does
engineering and compute."
```

---

**"What's catastrophic forgetting and how do you prevent it in BERT fine-tuning?"**

```
"Catastrophic forgetting: when a model trained on
task B loses performance on previously learned task A.

In BERT fine-tuning: if learning rate is too high
or training runs too long, the pre-trained weights
shift dramatically — BERT loses its language knowledge
and becomes overfit to the fine-tuning task. Accuracy
drops both on the task (overfitting) and on other tasks.

Prevention strategies:

1. Small learning rate (2e-5 to 5e-5)
   Weights move slowly, pre-trained values preserved.

2. Few epochs (2-4)
   Less opportunity to drift from pre-trained values.

3. Learning rate warmup
   Start from 0, ramp up — prevents early large updates
   from the randomly-initialized task head from
   propagating destructively through BERT layers.

4. Layer-wise learning rate decay
   Lower layers get smaller LR than upper layers.
   Lower layers: basic syntax, shouldn't change much.
   Upper layers: high-level semantics, can adapt more.
   Example: layer 1 LR = 1e-5, layer 12 LR = 5e-5.

5. Gradual unfreezing
   Fine-tune only task head first, then unfreeze
   upper BERT layers, then lower layers progressively."
```

## 12.8 The Two-Minute BERT Summary

Interviewers sometimes say: "Explain BERT to me in two minutes."

This is your answer:

```
"BERT is a bidirectional Transformer encoder pre-trained
on massive text using self-supervised learning.

Input text is tokenized with WordPiece into subwords,
then embedded by summing three learned embeddings —
token identity, position, and segment — producing
one 768-d vector per token.

These vectors flow through 12 Transformer blocks.
Each block has two sub-layers: multi-head self-attention,
which lets every token attend to every other token
simultaneously across 12 parallel heads — each learning
different relationship types like syntax, co-reference,
and semantics — followed by a per-token feed-forward
network that non-linearly transforms each token's
representation. Residual connections and layer
normalization wrap both sub-layers for training stability.

BERT is pre-trained on 3.3B words using masked language
modeling — predicting randomly masked tokens using
bidirectional context — which forces it to learn deep
language understanding without any labeled data.

The result: after 12 layers, each token has a rich,
contextual 768-d representation encoding its meaning
in the specific context of the full sequence.

For downstream tasks, you add a tiny task-specific head
— a linear layer for classification, per-token labels
for NER, or start/end vectors for QA — and fine-tune
all weights with a small learning rate on your labeled
data. Because BERT already understands language deeply,
you need surprisingly little labeled data to achieve
state-of-the-art performance."
```

## Chapter 12 Summary

The difference between a good candidate and a hired candidate is not knowing more facts — it's the ability to:

```
1. Connect concepts to each other
   ("residuals exist because of vanishing gradients,
    which exist because of chain rule multiplication,
    which is why 12-layer BERT couldn't train without them")

2. Reason about tradeoffs
   ("I'd use DistilBERT here because latency matters
    more than the 2% accuracy drop for this use case")

3. Handle failure cases
   ("if I saw accuracy drop in production, I'd first
    check data drift, then preprocessing mismatch...")

4. Know what you don't know
   ("I'm not certain about the exact SOP training details
    in ALBERT, but the motivation was...")
```

That last point is underrated. Google interviewers are testing your thinking process as much as your knowledge. Intellectual honesty about uncertainty, combined with solid first-principles reasoning, is more impressive than false confidence.
