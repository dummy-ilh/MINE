Recommendation system design

Two-stage retrieval + ranking · Netflix, YouTube, TikTok

Clarify scope first — always
  1. Business goal: engagement, revenue, retention, discovery, or safety?
  2. Scale: How many users? Items? QPS? (10M vs 1B changes everything)
  3. Latency budget: 50ms? 500ms? Real-time or batch?
  4. Feedback signal: explicit (ratings, thumbs) or implicit (clicks, watch time)?
  5. Cold start: constraints for new users? New items?
ML formulation

Frame as two tasks: (1) retrieve ~1000 candidates from millions, (2) rank precisely. Never run a heavy model over all items.

Retrieval: ANN search (FAISS/ScaNN) Ranking: pointwise / pairwise / listwise Output: ordered list of K items

Funnel: 10M items → 1000 (retrieval) → 200 (pre-ranking) → 10 (ranking) → response

Key components
  1. User embedding service — two-tower model, user vector cached in Redis with TTL
  2. Item embedding index — FAISS / ScaNN for ANN, pre-computed offline
  3. Feature store — Feast / Tecton; point-in-time correct to avoid leakage
  4. Ranking model — GBDT or DNN with cross features (DCN-v2, DIN, DIEN)
  5. Logging pipeline — Kafka → Flink → offline store; foundation of training data
  6. Training pipeline — daily/hourly retraining, shadow + canary deployment
User features
User ID embedding Demographics Interaction history (sequence) Session context (device, time of day) Long-term interests (30-day) Short-term intent (last 10 actions)
Item features
Item ID embedding Category / tags Content embedding (BERT / ViT) Global & regional popularity Freshness / recency Creator quality signals
Cross / interaction features
User × item affinity score Historical CTR on similar items Co-occurrence in sessions Category match score

Key trap: training-serving skew — always use the same feature store for both. Log features at serving time.

Tradeoffs to discuss in every interview
Latency vs quality
Heavier ranker = better results but slower. Cascade: cheap model first, expensive model on top-N only.
Explore vs exploit
ε-greedy, UCB, or Thompson sampling. Over-exploitation → filter bubble. Over-exploration → bad UX.
Freshness vs stability
Frequent retraining catches trends but risks regressions. Shadow deployment + canary rollout is key.
Personalization vs popularity
Cold-start users need popularity fallback. Blend: α·personal + (1−α)·global_popularity.
Recall vs precision
Retrieval optimizes recall (don't miss good items). Ranking optimizes precision (top spots must count).
High-signal interview questions
"How would you reduce latency from 500ms to 50ms?"
→ Pre-compute embeddings, quantize FAISS index, cache user vectors in Redis, cascade ranking, early exit.
"How do you handle the cold start problem?"
→ Content-based fallback, onboarding survey, transfer learning, explore new users more aggressively.
"Model has high AUC offline but business metrics drop in A/B test. Why?"
→ Metric misalignment, position bias, data leakage, distribution shift, novelty effect.
"How do you detect and fix training-serving skew?"
→ Log features at serving time, compare distributions, use same feature store for train and serve.
"How would you improve diversity without hurting relevance?"
→ Post-processing: MMR (maximal marginal relevance), DPP (determinantal point process), slot-based diversification.
